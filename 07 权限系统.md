## 1. 权限决策管道

每个工具调用在执行前都必须经过权限决策管道：

```
工具调用请求
    │
    ▼
hasPermissionsToUseTool（快速路径）
    │ 检查工具是否被禁用、是否在白名单中
    │
    ├─ 已允许 → 直接执行
    │
    ▼
选择处理器（根据执行上下文）
    │
    ├─ Interactive Handler     → 交互式REPL
    ├─ Swarm Worker Handler   → Swarm工作者
    └─ Coordinator Handler    → 协调器模式
    │
    ▼
PermissionDecision
    ├─ allow    → 执行工具
    ├─ deny     → 拒绝并记录
    └─ ask      → 需要用户确认
```

### 1.1 快速路径（hasPermissionsToUseTool）

```typescript
function hasPermissionsToUseTool(
  tool: Tool,
  input: any,
  context: ToolUseContext
): PermissionDecision | null {
  // 1. 工具被禁用
  if (!tool.isEnabled(context)) {
    return { decision: 'deny', reason: 'Tool is disabled' };
  }

  // 2. Bypass模式
  if (context.getAppState().permissionMode === 'bypass') {
    return { decision: 'allow' };
  }

  // 3. 工具自身的权限检查
  const toolPermission = tool.checkPermissions(input, context);
  if (toolPermission.decision !== 'continue') {
    return toolPermission;
  }

  // 4. 进入处理器路径
  return null;
}
```

## 2. ResolveOnce 模式

ResolveOnce 是权限系统中最精妙的并发控制模式，确保在多个异步响应源竞争时只有一个结果生效。

### 2.1 核心实现

```typescript
function createResolveOnce<T>(): ResolveOnce<T> {
  let claimed = false;    // 同步标志：是否已被声明
  let delivered = false;  // 异步标志：结果是否已送达

  return {
    claim(): boolean {
      if (claimed) return false;
      claimed = true;
      return true;
    },

    deliver(value: T): boolean {
      if (delivered) return false;
      delivered = true;
      resolve(value);
      return true;
    },

    isDelivered(): boolean {
      return delivered;
    }
  };
}
```

### 2.2 为什么需要两个标志

**`claimed`（同步）**：用于立即阻止其他竞争者进入处理逻辑。当一个响应源（如用户按键）成功 `claim()` 后，其他源即使同时触发也会被拒绝。

**`delivered`（异步）**：用于确保最终结果只被传递一次。`claim()` 和实际处理之间可能有异步操作（如网络请求），`delivered` 确保即使 `claim()` 后处理失败，不会有悬挂状态。

```
场景：用户按下Y的同时，自动分类器也返回了结果

用户按键:           claim() ✓ → 处理 → deliver(allow) ✓
自动分类器（同时）:  claim() ✗ → 不进入处理逻辑
```

如果没有 `claimed`，两个源可能都进入处理逻辑，导致竞态。

## 3. 交互式处理器（Interactive Handler）

交互式处理器是最复杂的路径，它同时赛跑6个并发源：

```typescript
async function interactivePermissionHandler(
  tool: Tool,
  input: any,
  context: ToolUseContext
): Promise<PermissionDecision> {
  const resolveOnce = createResolveOnce<PermissionDecision>();

  // 1. 用户对话框
  const dialogPromise = showPermissionDialog(tool, input)
    .then(result => resolveOnce.deliver(result));

  // 2. Bridge CCR响应
  const bridgePromise = waitForBridgeCCR(tool, input)
    .then(result => resolveOnce.deliver(result));

  // 3. Channel Relay（SDK通道）
  const channelPromise = waitForChannelRelay(tool, input)
    .then(result => resolveOnce.deliver(result));

  // 4. Hook系统
  const hookPromise = executePermissionHooks(tool, input)
    .then(result => resolveOnce.deliver(result));

  // 5. Bash自动分类器
  const classifierPromise = runBashClassifier(tool, input)
    .then(result => {
      // 分类器需要宽限期
      await sleep(GRACE_PERIOD_MS);
      resolveOnce.deliver(result);
    });

  // 6. Abort信号
  context.abortController.signal.addEventListener('abort', () =>
    resolveOnce.deliver({ decision: 'deny', reason: 'aborted' })
  );

  return resolveOnce.promise;
}
```

### 3.1 宽限期（Grace Period）

```typescript
const GRACE_PERIOD_MS = 200;

// 分类器结果需要经过宽限期
async function deliverClassifierResult(result: ClassificationResult) {
  await sleep(GRACE_PERIOD_MS);
  if (!resolveOnce.isDelivered()) {
    resolveOnce.deliver(result);
  }
}
```

#### 为什么需要200ms宽限期

当用户正在快速输入时，工具调用可能在用户的按键序列中间触发。如果分类器立即自动批准，用户可能：
- 正在按 N（拒绝）但来不及完成
- 正在打字，无意中触发了某个快捷键

200ms 的延迟给用户一个"反应窗口"——如果用户在这段时间内做出了选择，用户的选择优先于分类器。

### 3.2 对勾转换（Checkmark Transition）

```typescript
const FOCUSED_CHECKMARK_DURATION = 3000;   // 终端聚焦时3秒
const UNFOCUSED_CHECKMARK_DURATION = 1000; // 终端失焦时1秒

function showCheckmark(isFocused: boolean) {
  const duration = isFocused ? FOCUSED_CHECKMARK_DURATION : UNFOCUSED_CHECKMARK_DURATION;
  // 显示 ✓ 对勾图标
  setTimeout(() => {
    // 对勾消失，恢复正常UI
  }, duration);
}
```

当自动批准生效时显示对勾，让用户知道发生了自动决策。聚焦时显示更长（3s），因为用户可能正在关注；失焦时显示更短（1s），因为用户可能在做其他事。

## 4. Swarm Worker 处理器

Swarm Worker（团队工作者 Agent）不能直接显示对话框，所以处理逻辑不同：

```typescript
async function swarmWorkerPermissionHandler(
  tool: Tool,
  input: any,
  context: ToolUseContext
): Promise<PermissionDecision> {
  // 1. 先等待分类器结果（不竞赛，是阻塞等待）
  const classifierResult = await runBashClassifier(tool, input);
  if (classifierResult.decision !== 'unknown') {
    return classifierResult;
  }

  // 2. 转发给Leader
  const callback = registerMailboxCallback();
  await sendToLeader({
    type: 'permission_request',
    tool: tool.name,
    input,
    callbackId: callback.id,
  });

  // 3. 等待Leader的回复
  const leaderResponse = await Promise.race([
    callback.promise,
    timeout(30_000).then(() => null),
  ]);

  // 4. 如果Leader无响应，回退到交互式
  if (!leaderResponse) {
    return interactivePermissionHandler(tool, input, context);
  }

  return leaderResponse;
}
```

### 为什么 Swarm Worker 等待分类器而非竞赛

Worker 没有终端 UI，不能显示权限对话框。它必须先确定分类器的结论：
- 如果分类器说"允许"→ 直接执行
- 如果分类器说"未知"→ 转发给 Leader 决策
- 竞赛模式不适用，因为只有分类器和 Leader 两个源

### 为什么先注册回调再发送消息

```typescript
// 先注册回调
const callback = registerMailboxCallback();

// 再发送消息
await sendToLeader({ callbackId: callback.id });
```

如果先发送再注册，Leader 可能在回调注册之前就回复了（极端情况下），导致回复丢失。先注册确保无论 Leader 回复多快都能收到。

## 5. Coordinator 处理器

Coordinator 模式的处理更加保守，减少对用户的打扰：

```typescript
async function coordinatorPermissionHandler(
  tool: Tool,
  input: any,
  context: ToolUseContext
): Promise<PermissionDecision> {
  // 1. 先执行hooks（不竞赛，顺序执行）
  const hookResult = await executePermissionHooks(tool, input);
  if (hookResult.decision !== 'continue') {
    return hookResult;
  }

  // 2. 再执行分类器（不竞赛，顺序执行）
  const classifierResult = await runBashClassifier(tool, input);
  if (classifierResult.decision !== 'unknown') {
    return classifierResult;
  }

  // 3. 最后才显示对话框
  // 标记：告诉交互式处理器不要重新运行hooks和分类器
  context.setAppState(s => ({ ...s, coordinatorPreAwaited: true }));
  return interactivePermissionHandler(tool, input, context);
}
```

### 为什么顺序执行而非竞赛

Coordinator 的目标是最小化用户中断。顺序执行让自动化检查有充分时间完成：
- Hook 可能调用外部 API 进行策略检查
- 分类器需要时间分析命令
- 只有当所有自动化检查都无法做出决策时，才会打扰用户

### coordinatorPreAwaited 标志

当 Coordinator 已经运行了 hooks 和分类器后，回退到交互式处理器时，通过这个标志告诉交互式处理器不要重复运行这些检查。

## 6. 权限模式

| 模式 | 行为 | 使用场景 |
|------|------|---------|
| `ask` | 每次工具调用都询问用户 | 安全敏感环境 |
| `auto` | 自动批准匹配规则的操作 | 日常开发 |
| `default` | 使用配置文件的默认规则 | 默认模式 |
| `bypass` | 跳过所有权限检查 | 信任环境（危险） |
| `bubble` | 将权限决策冒泡给父Agent | Swarm Worker |
| `plan` | 只允许读操作，写操作需要计划审批 | 计划模式 |
| `ungated_auto` | 无门控的自动模式 | 特定企业配置 |

### 为什么有这么多模式

不同的部署场景需要不同的安全级别：
- **企业安全审计**：`ask` 模式，每个操作都有审计记录
- **个人开发者**：`auto` 模式，信任自动分类器
- **CI/CD**：`bypass` 模式，无人交互
- **多Agent工作者**：`bubble` 模式，交给协调者决策
- **代码审查**：`plan` 模式，只读安全

### 7.1 通配符匹配

```typescript
function matchRule(rule: PermissionRule, tool: string, input: any): boolean {
  // 工具名匹配
  if (rule.tool && !wildcardMatch(rule.tool, tool)) return false;

  // 命令匹配（Bash工具）
  if (rule.command && !wildcardMatch(rule.command, input.command)) return false;

  // 路径匹配（文件工具）
  if (rule.path && !wildcardMatch(rule.path, input.file_path)) return false;

  return true;
}
```

规则按定义顺序匹配，第一个匹配的规则生效（先匹配先生效）。

## 8. Bash 特定权限流程

BashTool 的权限检查是所有工具中最复杂的，详见 [[04 工具详解-BashTool]]。简要流程：

```
命令字符串
    │
    ▼
Parse（解析）
    │ 提取基础命令、参数、路径
    ▼
Semantics Check（语义检查）
    │ 理解命令的实际意图
    ▼
Security Checks（安全检查）
    │ 23项安全验证器
    ▼
Path Constraints（路径约束）
    │ 检查涉及的文件路径
    ▼
Command Operator Permissions（命令操作符权限）
    │ 管道、重定向、链接
    ▼
Permission Mode（权限模式）
    │ ask / auto / bypass
    ▼
Final Decision
```

## 10. Hook 系统

### 10.1 事件类型（20+）

| 类型 | 触发时机 | 用途 |
|------|---------|------|
| `pre_tool_use` | 工具执行前 | 权限检查、参数修改 |
| `post_tool_use` | 工具执行后 | 结果处理、审计 |
| `pre_api_call` | API调用前 | 请求修改、缓存 |
| `post_api_call` | API调用后 | 响应处理 |

| `subagent_end` | Agent结束 | 结果汇总 |

| `permission_request` | 权限请求 | 自定义权限逻辑 |
| `message_create` | 消息创建 | 内容过滤 |
| `tool_error` | 工具出错 | 错误处理 |
| `compact` | 上下文压缩 | 自定义压缩逻辑 |
| ... | ... | ... |

### 10.2 Hook 来源

```typescript
type HookSource =
  | { type: 'command'; command: string; args: string[] }     // Shell命令
  | { type: 'http'; url: string; method: string }             // HTTP请求
  | { type: 'prompt'; prompt: string; model: string }         // LLM提示
  | { type: 'sdk'; callback: HookCallback };                  // SDK回调
```

### 10.3 执行流程

```typescript
async function* executeHooks(
  eventType: string,
  data: any,
  hooks: Hook[]
): AsyncGenerator<HookEvent> {
  const matchingHooks = hooks.filter(h =>
    h.eventType === eventType &&
    (!h.ifCondition || evaluateCondition(h.ifCondition, data))
  );

  for (const hook of matchingHooks) {
    yield { type: 'hook_start', hook };

    try {
      switch (hook.source.type) {
        case 'command':
          yield* executeCommandHook(hook, data);
          break;
        case 'http':
          yield* executeHttpHook(hook, data);
          break;
        case 'prompt':
          yield* executePromptHook(hook, data);
          break;
        case 'sdk':
          yield* executeSdkHook(hook, data);
          break;
      }
    } catch (error) {
      yield { type: 'hook_error', hook, error };
    }

    yield { type: 'hook_end', hook };
  }
}
```

### 10.4 if-condition 匹配


### 10.5 async-rewakeable

```typescript
// Hook可以暂停并在稍后被唤醒
async function* executeAsyncHook(hook: Hook, data: any): AsyncGenerator<HookEvent> {
  const result = await startHookExecution(hook, data);

  if (result.status === 'pending') {
    // 暂停，等待外部信号唤醒
    yield { type: 'hook_suspended', hook, resumeToken: result.resumeToken };

    // 被唤醒后继续
    const finalResult = await waitForResume(result.resumeToken);
    yield { type: 'hook_resumed', hook, result: finalResult };
  }
}
```

## 11. 设计决策总结

| 决策 | 原因 |
|------|------|
| ResolveOnce 双标志 | 防止竞态的同步声明 + 异步送达 |
| 200ms宽限期 | 防止快速输入时的误批准 |
| Swarm Worker 等待不竞赛 | 无UI，需要确定性结果 |
| Coordinator 顺序执行 | 最小化用户中断 |
| 7种权限模式 | 覆盖不同部署和安全需求 |
| Hook系统可扩展 | 企业可添加自定义权限逻辑 |
| 权限日志扇出 | 多系统同时接收，用途不同 |
