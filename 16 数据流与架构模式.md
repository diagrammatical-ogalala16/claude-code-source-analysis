# 第16章 数据流与架构模式

> **作者**: [Wang Yanshu](https://catyans.github.io)

> 本章分析 Claude Code 中 12 种横切架构模式，这些模式跨越多个子系统，定义了数据如何流动、组件如何解耦、副作用如何管理。

## 1. 消息规范化边界

### 1.1 正向流（API 方向）

```
内部 Message 类型
    │ normalizeMessagesForAPI()
    ▼
API Message 格式
    │ （Anthropic/Bedrock/Vertex各有差异）
    ▼
HTTP 请求体
```

```typescript
function normalizeMessagesForAPI(messages: Message[]): APIMessage[] {
  return messages.map(msg => {
    // 移除内部字段
    const { internalId, toolUseContext, contextModifier, ...apiMsg } = msg;

    // 转换工具结果格式
    if (msg.role === 'tool') {
      return mapToolResultToToolResultBlockParam(msg);
    }

    return apiMsg;
  });
}
```

### 1.2 反向流（渲染方向）

```
API 响应
    │ 流式事件解析
    ▼
内部 Message 类型
    │ renderAndRun()
    ▼
React 组件树
    │ reconciler
    ▼
Screen 缓冲区
    │ 帧差分
    ▼
终端输出
```

### 为什么需要规范化边界

- **解耦**：内部消息格式可以包含元数据（internalId、contextModifier），API 格式不需要
- **多Provider适配**：Bedrock 和 Vertex 的消息格式与 Anthropic 略有不同
- **双向一致**：规范化保证内部格式是唯一的真实来源

## 2. Context Modifier 模式

### 2.1 机制

```typescript
// 工具执行返回contextModifier
interface ToolResult {
  output: any;
  contextModifier?: (messages: Message[]) => Message[];
}

// 修改器被排队，不立即执行
const queuedContextModifiers: Record<string, ContextModifier> = {};

// 工具执行完成后
function onToolComplete(toolUseId: string, result: ToolResult) {
  if (result.contextModifier) {
    queuedContextModifiers[toolUseId] = result.contextModifier;
  }
}

// 所有工具执行完成后，按顺序应用
function applyContextModifiers(messages: Message[]): Message[] {
  const sortedIds = Object.keys(queuedContextModifiers).sort();
  let result = messages;

  for (const id of sortedIds) {
    result = queuedContextModifiers[id](result);
  }

  return result;
}
```

### 2.2 实际用例

```typescript
// FileEditTool 的 contextModifier
function fileEditContextModifier(messages: Message[]): Message[] {
  // 更新消息中缓存的文件内容
  return messages.map(msg => {
    if (msg.cachedFileContent?.[editedPath]) {
      return {
        ...msg,
        cachedFileContent: {
          ...msg.cachedFileContent,
          [editedPath]: newContent,
        },
      };
    }
    return msg;
  });
}
```

### 为什么排队而非立即应用

- **并发安全**：多个只读工具可能并发执行，它们的 modifier 如果立即应用会相互干扰
- **确定性顺序**：按 toolUseId 排序保证了应用顺序一致
- **事务性**：所有工具都完成后一次性应用，避免中间状态

## 3. 投机执行叠加（Speculative Execution Overlay）

### 3.1 机制

```typescript
// 在模型流式输出文件写入指令时
function onStreamingToolCall(toolName: string, partialInput: any) {
  if (toolName === 'FileWrite' && partialInput.file_path && partialInput.content) {
    // 预测：这个文件将被写入
    applySpeculativeWrite(partialInput.file_path, partialInput.content);
  }
}

// 如果预测正确
function onToolExecutionSuccess(toolName: string, input: any) {
  if (matchesSpeculativeWrite(toolName, input)) {
    // 无缝过渡——文件已经"写入"了
    confirmSpeculativeWrite(input.file_path);
    logTimeSaved();
  }
}

// 如果预测错误
function onToolExecutionFailure(toolName: string, input: any) {
  // 回滚投机写入
  rollbackSpeculativeWrite(input.file_path);
}
```

### 3.2 "节省时间"计算

```typescript
function calculateTimeSaved(fileSize: number, toolLatency: number): number {
  // 投机执行期间用户已经看到了文件变更
  // 实际工具执行时间就是"节省"的时间
  return toolLatency; // 通常50-200ms
}
```

### 为什么投机执行安全

- **只影响 UI 显示**：投机写入不会实际修改磁盘上的文件
- **可回滚**：错误时完全撤销
- **仅用于高确定性操作**：只有 FileWrite 这类简单工具才被投机执行

## 4. AsyncGenerator 流式模式

### 4.1 系统中的使用位置

```
query loop    →  AsyncGenerator<QueryEvent>
tool execution →  AsyncGenerator<ToolProgress, ToolResult>
hook execution →  AsyncGenerator<HookEvent>
API streaming  →  AsyncGenerator<StreamEvent>
```

### 4.2 统一优势

| 特性 | 说明 |
|------|------|
| 原生取消 | `generator.return()` 在任何 yield 点取消 |
| 背压 | 消费者控制 `next()` 的调用频率 |
| 渐进渲染 | 每个 yield 都可以触发 UI 更新 |
| 组合性 | `yield*` 可以委托给子生成器 |
| 状态封装 | 局部变量就是状态 |

### 4.3 组合示例

```typescript
async function* queryLoop(): AsyncGenerator<QueryEvent> {
  // 委托给API流
  const response = yield* streamApiCall();

  // 委托给工具执行
  for (const toolCall of response.toolCalls) {
    yield* executeTool(toolCall);
  }

  // 委托给钩子执行
  yield* runStopHooks();
}
```

## 5. 缓存上下文构建器

### 5.1 getSystemContext

```typescript
let cachedSystemContext: SystemContext | null = null;
let systemContextSessionId: string | null = null;

async function getSystemContext(): Promise<SystemContext> {
  if (cachedSystemContext && systemContextSessionId === currentSessionId) {
    return cachedSystemContext;
  }

  // 昂贵操作
  const [gitStatus, claudeMd, gitignore, projectStructure] = await Promise.all([
    exec('git status --porcelain'),      // ~100ms
    readClaudeMd(),                       // ~50ms
    readGitignore(),                      // ~20ms
    scanProjectStructure(),               // ~200ms
  ]);

  cachedSystemContext = { gitStatus, claudeMd, gitignore, projectStructure };
  systemContextSessionId = currentSessionId;
  return cachedSystemContext;
}
```

### 5.2 getUserContext

```typescript
let cachedUserContext: UserContext | null = null;

async function getUserContext(): Promise<UserContext> {
  if (cachedUserContext) return cachedUserContext;

  const [userName, userEmail, osInfo] = await Promise.all([
    exec('git config user.name'),
    exec('git config user.email'),
    getOSInfo(),
  ]);

  cachedUserContext = { userName, userEmail, osInfo };
  return cachedUserContext;
}
```

### 为什么按会话缓存

- `git status` 约 100ms，每轮调用会严重影响响应速度
- 项目结构在一个会话内很少变化
- 用户上下文（git 用户名等）在整个进程生命周期内不变

## 6. 两阶段渲染

### 6.1 逻辑阶段（query.ts）

```typescript
// 产出的事件不包含任何渲染信息
yield {
  type: 'assistant_message',
  content: response.content,
  model: response.model,
  usage: response.usage,
};
```

### 6.2 视觉阶段（components/）

```typescript
// React组件消费事件进行渲染
function AssistantMessage({ event }: Props) {
  return (
    <Box>
      <ModelBadge model={event.model} />
      <MarkdownRenderer content={event.content} />
      <TokenCounter usage={event.usage} />
    </Box>
  );
}
```

### 为什么分离

同一套查询逻辑（逻辑阶段）服务于三种不同的渲染目标：
- **REPL**：React 组件渲染到终端
- **SDK/Print**：结构化 JSON 输出
- **Transcript**：纯文本记录

如果逻辑和渲染耦合，就需要三套查询逻辑。

## 7. 惰性模块加载模式

### 7.1 工厂函数断循环依赖

```typescript
// 避免循环 import
let _TeamCreateTool: typeof import('./team/create').TeamCreateTool | null = null;

function getTeamCreateTool() {
  if (!_TeamCreateTool) {
    _TeamCreateTool = require('./team/create').TeamCreateTool;
  }
  return _TeamCreateTool;
}
```

### 7.2 为什么不用动态 import()

- `require()` 是同步的，可以在非异步上下文中使用
- 工厂函数只在首次调用时执行 `require()`，后续使用缓存
- 动态 `import()` 返回 Promise，需要所有调用者支持异步

## 8. Feature Flag DCE 模式

```typescript
// 编译时分支
const MyModule = feature('X') ? require('./x.js').X : null;

// 运行时安全使用
function maybeUseX() {
  if (MyModule) {
    return new MyModule();
  }
  return fallback();
}
```

编译器将 `feature('X')` 替换为 `false`：
```typescript
const MyModule = false ? require('./x.js').X : null;
// 简化为：
const MyModule = null;
// require('./x.js') 被DCE移除
```

## 9. Fire-and-Forget 分析

```typescript
// 遥测不应阻塞任何操作
function logAnalytics(event: AnalyticsEvent): void {
  // 不await，不catch
  queueMicrotask(async () => {
    try {
      await sendToAnalytics(event);
    } catch {
      // 静默失败
    }
  });
}
```

### 为什么用 queueMicrotask

- `setTimeout(fn, 0)`：会在下一个宏任务执行，可能延迟较长
- `queueMicrotask(fn)`：在当前微任务队列末尾执行，延迟最小
- `Promise.resolve().then(fn)`：与 queueMicrotask 类似，但多一层 Promise 包装

## 10. 权限上下文隔离

三种处理器路径完全独立：

```
                    ┌─ Interactive Handler
                    │  （6路竞赛）
权限请求 ──────────┤
                    ├─ SwarmWorker Handler
                    │  （等待→转发→回退）
                    │
                    └─ Coordinator Handler
                       （顺序：hooks→分类器→对话框）
```

每种处理器有自己的状态和逻辑，不共享中间结果。这避免了一种处理器的 bug 影响另一种。

## 11. Bridge 依赖注入

```typescript
// 可测试的工厂函数
function createBridgeApiClient(deps: BridgeDeps): BridgeApiClient {
  return {
    async poll() {
      const response = await deps.httpClient.get(deps.pollUrl);
      return response.data;
    },

    async ack(workId: string) {
      await deps.httpClient.post(`${deps.ackUrl}/${workId}`);
    },
  };
}

function createTokenRefreshScheduler(deps: TokenDeps): TokenScheduler {
  return {
    schedule(token: string) {
      const expiry = deps.decodeJwt(token).exp;
      deps.setTimeout(() => this.refresh(), expiry - deps.now() - 5 * 60 * 1000);
    },

    async refresh() {
      const newToken = await deps.fetchToken();
      deps.onTokenRefreshed(newToken);
    },
  };
}
```

### 为什么用依赖注入

Bridge 组件需要在单元测试中运行，但它们依赖：
- HTTP 客户端（需要 mock 网络）
- JWT 解码器（需要控制时间）
- setTimeout（需要控制时间流逝）

通过将这些依赖注入而非硬编码，测试可以提供 mock 实现。

## 12. 批处理与并发控制

### 12.1 三层并发模型

```
Level 1: 只读工具并行（max 10）
  ├─ GrepTool ──┐
  ├─ GlobTool ──┤ Promise.all（带信号量）
  └─ FileRead ──┘

Level 2: 写入工具串行
  ├─ FileWrite → 等待完成
  ├─ BashTool  → 等待完成
  └─ FileEdit  → 等待完成

Level 3: Context Modifier 串行
  └─ 按toolUseId顺序逐一应用
```

### 12.2 为什么 max=10

- 文件描述符限制：每个 ripgrep 调用打开多个 FD
- 进程数量：BashTool 的每个调用都是一个子进程
- I/O 带宽：SSD 上 10 个并发读取的吞吐量接近峰值

## 13. 设计决策总结

| 模式 | 用途 | 关键好处 |
|------|------|---------|
| 消息规范化边界 | API/内部格式转换 | 解耦、多Provider适配 |
| Context Modifier | 工具修改上下文 | 并发安全、确定性顺序 |
| 投机执行 | 流式预测写入 | 减少感知延迟 |
| AsyncGenerator | 统一流式接口 | 取消、背压、组合 |
| 缓存上下文 | 避免重复计算 | git status 100ms→0ms |
| 两阶段渲染 | 逻辑/视觉分离 | 一套逻辑三种输出 |
| 惰性加载 | 断循环依赖 | 启动加速 |
| DCE模式 | 编译时裁剪 | 安全、体积优化 |
| Fire-and-forget | 异步分析 | 不阻塞用户体验 |
| 权限隔离 | 三种处理路径 | 故障隔离 |
| 依赖注入 | 可测试Bridge | 纯mock测试 |
| 三层并发 | 读并行写串行 | 安全+性能平衡 |
