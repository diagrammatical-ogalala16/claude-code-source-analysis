# 第11章 记忆系统

> **作者**: [Wang Yanshu](https://catyans.github.io)

> 本章分析 memdir/（8个文件）的记忆存储、召回和团队共享机制。

## 1. 记忆文件格式

每个记忆文件采用 YAML frontmatter + Markdown body 的格式：

```markdown
---
name: "API设计偏好"
description: "用户对REST API设计的个人偏好"
type: user
---

## 偏好

- 使用kebab-case作为URL路径
- 响应始终包含request_id
- 错误响应使用RFC 7807格式
- 分页使用cursor-based而非offset-based

## 原因

用户在多个项目中一致使用这些模式，且在反馈中明确表示偏好。
```

### 1.1 Frontmatter 解析

```typescript
function parseFrontmatter(content: string): MemoryMeta {
  const lines = content.split('\n');
  const first30Lines = lines.slice(0, 30); // 只读前30行

  let inFrontmatter = false;
  let yamlContent = '';

  for (const line of first30Lines) {
    if (line.trim() === '---') {
      if (inFrontmatter) {
        // frontmatter结束
        break;
      } else {
        inFrontmatter = true;
        continue;
      }
    }
    if (inFrontmatter) {
      yamlContent += line + '\n';
    }
  }

  return yaml.parse(yamlContent) as MemoryMeta;
}
```

### 为什么只读前30行

- 记忆文件可能很长（包含详细的参考资料）
- Frontmatter 通常在前 5-10 行
- 限制为 30 行是一个安全余量，避免读取整个大文件只为了解析元数据
- 这在批量扫描数百个记忆文件时带来显著的性能提升

## 2. 四种记忆类型

| 类型 | 英文 | 用途 | 内容结构 |
|------|------|------|---------|
| 用户记忆 | user | 角色、目标、偏好 | 直接描述 |
| 反馈记忆 | feedback | 什么该避免、什么该重复 | Why + How to apply |
| 项目记忆 | project | 决策、上下文、约定 | Why + How to apply |
| 参考记忆 | reference | 外部资源指针 | URL + 摘要 |

用户记忆记录的是跨项目、跨会话不变的个人特征。Claude 可以据此调整回答的技术深度和风格。

### 2.2 反馈记忆（Feedback）

```markdown
---
name: "代码审查反馈"
description: "用户对代码生成质量的反馈"
type: feedback
---

## 避免

- 不要在函数中使用超过3层嵌套
  - **Why**: 用户多次要求扁平化嵌套结构
  - **How to apply**: 使用早返回、guard clause

## 重复

- 每个函数添加JSDoc注释
  - **Why**: 用户在3个不同项目中都要求了这一点
  - **How to apply**: 包含@param、@returns、@throws
```

反馈记忆的 Why + How to apply 结构使得未来的 Claude 实例能够判断边界情况——不是盲目遵循规则，而是理解规则的意图后做出判断。

### 2.3 项目记忆（Project）

```markdown

name: "数据库迁移策略"
description: "本项目的数据库迁移约定"

---

## 决策

- 使用 Prisma 进行数据库迁移

- 迁移文件命名：YYYYMMDD_description
- 每次迁移只做一件事
```

## 3. MEMORY.md 入口点

### 3.1 格式和限制

```typescript
const MEMORY_MD_MAX_LINES = 200;
const MEMORY_MD_MAX_BYTES = 25_000; // 25KB

function truncateMemoryMd(content: string): string {
  const lines = content.split('\n');

  // 先按行截断
  if (lines.length > MEMORY_MD_MAX_LINES) {
    content = lines.slice(0, MEMORY_MD_MAX_LINES).join('\n');
  }

  // 再按字节截断（在最后一个换行处截断，保持行完整性）
  if (Buffer.byteLength(content) > MEMORY_MD_MAX_BYTES) {
    const buffer = Buffer.from(content);
    const truncated = buffer.slice(0, MEMORY_MD_MAX_BYTES);
    const lastNewline = truncated.lastIndexOf(0x0a); // '\n'
    content = truncated.slice(0, lastNewline).toString();
  }

  return content;
}
```

### 为什么两级截断

1. **先行截断**：200 行是一个语义边界——每行通常是一个记忆条目的引用
2. **再字节截断**：某些行可能很长（如包含长 URL 的参考记忆），需要第二层保护
3. **在最后一个换行处截断**：确保不会截断在行中间，保持输出的可读性

### 3.2 为什么限制 200 行 / 25KB

MEMORY.md 在每个 API 调用中都作为系统提示的一部分发送给模型。过大的 MEMORY.md 会：
- 消耗大量 token 预算（25KB ≈ 6K-8K tokens）
- 降低模型对其他上下文信息的注意力
- 增加每次 API 调用的成本

200 行 / 25KB 是一个经验平衡点：足够存储关键记忆，不至于挤压工作上下文。

## 4. 记忆召回机制

### 4.1 召回流程

```
用户发送消息
    │
    ▼
扫描记忆文件（scanMemoryFiles）
    │ 读取所有记忆文件的头部信息
    ▼
格式化记忆清单（formatMemoryManifest）
    │ 每个文件一行：名称 + 描述 + 类型
    ▼
侧查询 Sonnet（sideQuery）
    │ 发送用户消息 + 记忆清单
    │ 要求返回相关记忆文件名
    ▼
过滤有效文件名（validFilenames）
    │ 确保返回的文件名存在
    ▼
过滤已展示（alreadySurfaced）
    │ 避免重复展示同一记忆
    ▼
加载记忆内容
    │ 读取选中的记忆文件全文
    ▼
注入到上下文
```

### 4.2 记忆扫描

```typescript
async function scanMemoryFiles(memoryDir: string): Promise<MemoryFileInfo[]> {
  const files: MemoryFileInfo[] = [];
  const entries = await fs.readdir(memoryDir);

  // 限制扫描数量
  const maxFiles = 200;
  let count = 0;

  for (const entry of entries) {
    if (count >= maxFiles) break;
    if (!entry.endsWith('.md')) continue;

    const filePath = path.join(memoryDir, entry);
    const stat = await fs.stat(filePath);

    // 读取frontmatter（只读前30行）
    const content = await readFileInRange(filePath, 0, 30);
    const meta = parseFrontmatter(content);

    files.push({
      path: filePath,
      name: meta.name,
      description: meta.description,
      type: meta.type,
      mtimeMs: stat.mtimeMs, // 用于新鲜度指示
    });

    count++;
  }

  return files;
}
```

### 4.3 记忆清单格式

```typescript
function formatMemoryManifest(files: MemoryFileInfo[]): string {
  return files
    .map(f => `${f.path} | ${f.type} | ${f.name}: ${f.description}`)
    .join('\n');
}
```

每个文件压缩为一行，让 Sonnet 可以快速浏览所有可用记忆，而不需要读取完整内容。

### 4.4 Sonnet 侧查询

```typescript
async function recallMemories(
  userMessage: string,
  manifest: string,
  alreadySurfaced: Set<string>
): Promise<string[]> {
  const response = await sideQuery({
    model: 'claude-sonnet',
    messages: [{
      role: 'user',
      content: `Given this user message:
"${userMessage}"

And these available memory files:
${manifest}

Which files are most relevant? Return a JSON array of file paths.
Do NOT include files already shown: ${[...alreadySurfaced].join(', ')}`,
    }],
    responseFormat: { type: 'json_object' },
  });

  const parsed = JSON.parse(response.content);
  return parsed.files.filter((f: string) => validFilenames.has(f));
}
```

### 为什么用 Sonnet 而非嵌入搜索

| 对比 | 嵌入搜索 | Sonnet 召回 |
|------|---------|-------------|
| 语义理解 | 表面相似度 | 深层语义关联 |
| 模糊查询 | 差 | 好 |
| 上下文感知 | 无 | 理解对话上下文 |
| 成本 | 低 | 中（一次侧查询） |
| 准确率 | 中 | 高 |

关键优势：Sonnet 能理解"用户正在处理 API 错误处理"然后关联到"API 设计偏好"记忆，而嵌入搜索可能因为关键词不匹配而错过这个关联。

## 5. 记忆年龄追踪

```typescript
function getMemoryAgeIndicator(mtimeMs: number): string {
  const ageHours = (Date.now() - mtimeMs) / (1000 * 60 * 60);

  if (ageHours < 24) return '🟢 最近更新';    // <1天
  if (ageHours < 168) return '🟡 本周更新';    // <7天
  if (ageHours < 720) return '🟠 本月更新';    // <30天
  return '🔴 较旧';                            // >30天
}
```

年龄指示帮助 Sonnet 在选择记忆时考虑时效性——较新的记忆通常更相关。

## 7. 什么不该保存

系统明确定义了不应存储为记忆的内容：

| 不保存 | 原因 |
|--------|------|
| 代码模式 | 可从代码库中推导 |
| Git 历史 | 通过 `git log` 可获取 |
| 调试方案 | 应该在代码中注释 |
| CLAUDE.md 中已有的内容 | 避免重复 |
| 临时任务细节 | 会过时，浪费空间 |

```typescript
function shouldSaveAsMemory(content: string, claudeMd: string): boolean {
  // 检查是否与CLAUDE.md重复
  if (isDuplicate(content, claudeMd)) return false;

  // 检查是否是临时内容
  if (isEphemeral(content)) return false;

  // 检查是否可从代码推导
  if (isDerivableFromCode(content)) return false;

  return true;
}
```

## 8. 记忆提取（自动保存）

在每轮对话结束后，系统会自动分析是否有值得保存的记忆：

```typescript
async function extractMemories(messages: Message[]): Promise<MemoryCandidate[]> {
  // 使用LLM分析最近的对话
  const candidates = await analyzeForMemories(messages);

  // 过滤已存在的记忆
  const existing = await scanMemoryFiles(MEMORY_DIR);
  const novel = candidates.filter(c =>
    !existing.some(e => isSimilar(c, e))
  );

  return novel;
}
```

这是一个 fire-and-forget 操作——不阻塞用户的下一轮输入。

## 9. 设计决策总结

| 决策 | 原因 |
|------|------|
| 文件式存储（非数据库） | 可跨会话、可手动检查/编辑、无迁移需求 |
| Sonnet 召回（非嵌入搜索） | 深层语义理解、处理模糊查询 |
| 200行/25KB 上限 | MEMORY.md 在每次API调用中加载，必须精简 |
| 先行截断再字节截断 | 保持自然行边界 |
| 4种类型 | 引导保存内容的方向，避免重复/陈旧 |
| Why + How to apply | 让未来的Claude判断边界情况而非盲目遵循 |
| 团队记忆隔离目录 | 共享但与个人记忆分离 |
| Fire-and-forget 提取 | 不阻塞轮次完成 |
