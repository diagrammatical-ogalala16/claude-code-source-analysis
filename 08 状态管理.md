## 2. Store 实现

### 2.1 createStore（35行核心实现）

```typescript
function createStore<T>(initialState: T): Store<T> {
  let state = initialState;
  const listeners = new Set<(state: T) => void>();

  return {
    getState(): T {
      return state;
    },

    setState(updater: (prev: T) => T): void {
      const nextState = updater(state);

      // Object.is 引用相等检查
      if (Object.is(state, nextState)) return;

      state = nextState;

      // 通知所有监听者
      for (const listener of listeners) {
        listener(state);
      }
    },

    subscribe(listener: (state: T) => void): () => void {
      listeners.add(listener);
      return () => listeners.delete(listener);
    },
  };
}
```

### 2.2 为什么不用 Redux/Zustand

Claude Code 没有使用任何主流状态管理库，原因：

| 对比项 | Redux/Zustand | createStore |
|-------|---------------|-------------|
| API 复杂度 | 中间件、action、reducer | 3 个方法 |
| 外部同步 | 需要中间件 | 直接回调 |
| React 耦合 | 深度耦合 | React 无关 |
| 包大小 | 10-30KB | ~35 行 |
| 序列化 | 需要可序列化 | 任意类型 |

关键差异：
- Claude Code 的状态同步发生在 React 外部（Bridge 通信、CCR 协议、文件写入），使用中间件会增加不必要的抽象层。
- Store 被用在非 React 上下文中（权限处理器、工具执行器），不应依赖 React 的调度机制。
- `Object.is` 引用相等检查对 Claude Code 的更新模式足够——每次 `setState` 都创建新的状态对象。

## 3. AppStateProvider

```typescript
function AppStateProvider({ children, store }: Props) {
  const [state, setState] = React.useState(store.getState());

  React.useEffect(() => {
    // 订阅store变化
    const unsubscribe = store.subscribe(nextState => {
      setState(nextState);
    });

    // 同步设置变化
    return () => {
      unsubscribe();
    };
  }, [store]);

  // 在远程设置加载后禁用bypass模式
  React.useEffect(() => {
    if (state.settingsSources.includes('remote')) {
      if (state.isBypassMode) {
        store.setState(s => ({ ...s, isBypassMode: false }));
      }
    }
  }, [state.settingsSources]);

  return (
    <AppStateContext.Provider value={{ state, store }}>
      {children}
    </AppStateContext.Provider>
  );
}
```

### 远程设置加载后禁用 bypass

这是一个安全措施：当远程管理设置（企业策略）加载成功后，bypass 模式被强制关闭。企业管理员可以通过远程设置来控制用户是否可以使用 bypass 模式。

## 4. onChangeAppState（6个同步点）

当 AppState 变化时，某些字段需要同步到外部系统：

```typescript
function onChangeAppState(prev: AppState, next: AppState): void {
  // 1. 权限模式 → CCR/SDK
  if (prev.permissionMode !== next.permissionMode) {
    notifyCCR({ type: 'permission_mode_changed', mode: next.permissionMode });
    notifySDK({ type: 'permission_mode_changed', mode: next.permissionMode });
  }

  // 2. 模型变更 → settings.json
  if (prev.currentModel !== next.currentModel) {
    updateSettingsFile({ model: next.currentModel });
  }

  // 3. 视图展开 → globalrc.json
  if (prev.viewExpanded !== next.viewExpanded) {
    updateGlobalRC({ viewExpanded: next.viewExpanded });
  }

  // 4. Verbose → globalrc.json
  if (prev.verbose !== next.verbose) {
    updateGlobalRC({ verbose: next.verbose });
  }

  // 5. Tmux面板 → globalrc.json
  if (prev.tmuxPanelId !== next.tmuxPanelId) {
    updateGlobalRC({ tmuxPanelId: next.tmuxPanelId });
  }

  // 6. 设置变更 → 认证缓存失效 + 环境变量
  if (prev.settings !== next.settings) {
    invalidateAuthCache();
    syncEnvironmentVariables(next.settings);
  }
}
```

### 为什么集中在一个 onChange 回调

单一的 `onChange` 回调是有意的设计，相比分散在各处的 `subscribe` 调用：
- **可审计**：所有外部同步都在一个文件中，容易审查
- **防遗漏**：新增同步逻辑时，不会忘记某个 callsite
- **有序执行**：6 个同步点按顺序执行，可以控制依赖关系（如设置变更导致缓存失效必须在环境变量同步之前）

## 5. DeepImmutable 类型

```typescript
type DeepImmutable<T> = T extends Function
  ? T
  : T extends Map<infer K, infer V>
    ? ReadonlyMap<DeepImmutable<K>, DeepImmutable<V>>
    : T extends Set<infer U>
      ? ReadonlySet<DeepImmutable<U>>
      : T extends Array<infer U>
        ? ReadonlyArray<DeepImmutable<U>>
        : T extends object
          ? { readonly [K in keyof T]: DeepImmutable<T[K]> }
          : T;
```

所有通过 `getState()` 获取的状态都被标记为 `DeepImmutable`，TypeScript 编译器会阻止直接修改属性：

```typescript
const state = store.getState();
state.verbose = true;           // TS错误：readonly属性
state.tasks.push(newTask);      // TS错误：ReadonlyArray没有push方法

// 正确方式：通过setState
store.setState(s => ({
  ...s,
  verbose: true,
  tasks: [...s.tasks, newTask],
}));
```

### 例外情况

四个字段不受 DeepImmutable 约束：

```typescript
// 这些字段包含函数或Map，freeze会有性能损失
tasks: Task[];                    // Task包含回调函数
agentNameRegistry: Map<string, string>;  // Map需要mutable操作
replContext: REPLContext;          // 包含readline实例
teamContext: TeamContext;          // 包含网络连接
```

**为什么这些是例外**：
- `tasks`：Task 对象包含回调函数（onComplete、onError），冻结函数没有意义
- `agentNameRegistry`：频繁的 set/get 操作，ReadonlyMap 会强制每次创建新 Map
- `replContext`：包含 readline 接口实例，是固有的可变对象
- `teamContext`：包含网络连接对象，不能被冻结

## 7. 设计决策总结

| 决策 | 原因 |
|------|------|
| 35行自定义Store | 最小API，React无关，直接回调同步 |
| 150+字段单一状态树 | 6种执行模式的统一状态视图 |
| DeepImmutable | 编译时防止意外修改 |
| 4个可变例外 | 函数和Map无法高效冻结 |
| 单一onChange回调 | 集中审计，防遗漏，有序执行 |
| Object.is相等检查 | 对函数式更新模式足够 |
| AppStateProvider同步 | React生命周期与Store变化桥接 |
