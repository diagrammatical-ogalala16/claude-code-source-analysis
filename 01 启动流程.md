# 第1章 启动流程

> **作者**: [Wang Yanshu](https://catyans.github.io)

> 本章深入分析 Claude Code 从 `main.tsx`（约3,500行）到完全就绪的完整启动管线。启动流程是理解整个系统的关键入口点，涉及模块级副作用、并行预取、Commander.js 命令系统和6种执行模式的路由。

## 1. 模块级副作用（Module-Level Side Effects）

在 `main.tsx` 文件的最顶部，甚至在任何 `import` 语句之前，就有几行关键的模块级代码被执行。这些代码利用了 JavaScript 模块系统的特性——模块在首次 `import` 时会被求值，且副作用代码会立即执行。

### 1.1 profileCheckpoint

```typescript
// main.tsx 顶部，所有import之前
const profileCheckpoint = createProfileCheckpoint();
```

`profileCheckpoint` 用于性能剖析，记录启动过程中各个阶段的耗时。它在模块级创建，确保能捕获最早的时间戳。这是一个高精度计时器，使用 `performance.now()` 而非 `Date.now()`，因为后者精度只有毫秒级且受系统时钟调整影响。

### 1.2 startMdmRawRead

```typescript
const mdmReadPromise = startMdmRawRead();
```

MDM（Mobile Device Management）读取是企业部署场景的关键路径。此函数会根据平台差异：
- **macOS**：spawns `plutil` 子进程读取 `/Library/Managed Preferences/` 下的 plist 配置
- **Windows**：spawns `reg query` 子进程读取注册表键值
- **Linux**：读取 `/etc/claude-code/` 下的配置文件

这些读取操作涉及子进程创建，耗时约 50-135ms。通过在模块级启动，这个 I/O 操作可以与后续的 `import` 语句（模块解析和编译）并行执行。

### 1.3 startKeychainPrefetch

```typescript
const keychainPromise = startKeychainPrefetch();
```

在 macOS 上，此函数并行发起两个 Keychain 读取：
1. **OAuth Token 读取**：从 macOS Keychain 中读取已存储的 OAuth 令牌
2. **Legacy API Key 读取**：从 Keychain 中读取旧版直接 API 密钥

macOS Keychain 的读取操作是同步的阻塞调用（通过 `security find-generic-password` 命令），每次约 30-40ms。预取这些凭证可以在后续认证流程中直接使用缓存结果。

### 为什么这样设计

模块级副作用是一个刻意的设计选择，而非偶然：

- **并行预取节省约65ms**：macOS Keychain 读取是同步 spawn 操作，非常慢。在模块级启动后，这些 I/O 操作与 Bun 的模块解析并行进行。
- **MDM 读取重叠模块求值**：MDM 配置读取约 135ms，而整个模块 import 链的求值也需要类似的时间，两者几乎完全重叠。
- **零成本抽象**：如果这些值最终不被使用（例如非 macOS 平台），Promise 只是被忽略，没有额外开销。

## 2. main() 入口函数

模块级副作用启动后，`main()` 函数是实际的入口点：

### 2.1 信号处理（SIGINT Dual Handling）

```typescript
function main() {
  if (isPrintMode) {
    // Print/SDK模式：异步abort → graceful shutdown
    process.on('SIGINT', async () => {
      abortController.abort();
      await gracefulShutdown();
      process.exit(130);
    });
  } else {
    // 交互模式：立即exit(0)，让Ink的process.exit hooks清理
    process.on('SIGINT', () => {
      process.exit(0);
    });
  }
}
```

双模式 SIGINT 处理的原因：
- **Print 模式**需要异步中止当前 API 请求，等待流式响应停止，然后优雅关闭——因为输出可能被管道传给其他程序，突然中断会导致下游程序收到不完整数据。
- **交互模式**需要立即退出，因为用户按 Ctrl+C 期望即时响应。Ink 框架已经注册了 `process.exit` 钩子来清理终端状态（恢复光标、退出 alt screen 等），所以直接 `exit(0)` 是安全的。

### 2.2 早期 argv 解析

在 Commander.js 的完整解析之前，`main()` 会对 `process.argv` 做一轮快速的手动解析，处理几种特殊场景：

```typescript
// Deep Link 处理
if (argv.includes('cc://') || argv.includes('--handle-uri')) {
  return handleDeepLink(argv);
}

// SSH 远程启动
if (argv.includes('ssh')) {
  return handleSSHMode(argv);
}

// Assistant 命令暂存
if (argv.includes('assistant')) {
  stashedAssistantArgs = extractAssistantArgs(argv);
}
```

这些特殊路径需要在 Commander.js 解析之前处理，因为：
- `cc://` deep link 有自己的 URI 解析逻辑，不符合 Commander.js 的参数格式
- SSH 模式需要在建立远程连接后才能进行常规初始化
- Assistant 参数需要暂存，在后续认证完成后才使用

### 2.3 交互模式检测

```typescript
const isNonInteractive = hasPrintFlag || hasInitOnlyFlag || hasSdkUrl || !process.stdout.isTTY;
```

四个条件中任一为真即进入非交互模式：
- `hasPrintFlag`：`--print` 或 `-p` 标志，单次查询模式
- `hasInitOnlyFlag`：`--init-only`，仅初始化不运行
- `hasSdkUrl`：通过 SDK URL 连接，程序化访问
- `!process.stdout.isTTY`：输出被管道重定向，无终端

## 4. 急切设置加载（Eager Settings Loading）

```typescript
// 在 init() 之前就解析 --settings 和 --setting-sources
const earlySettings = parseEarlySettings(argv);
if (earlySettings) {
  applySettingsOverrides(earlySettings);
}
```

设置必须在 `init()` 之前加载，原因是：
- `init()` 中的认证流程可能需要读取自定义设置（如自定义 API 端点）
- 企业部署场景下，MDM 管理的设置需要最早生效
- `--settings` 允许指定完整的设置文件路径，`--setting-sources` 允许指定设置来源优先级

## 5. Commander.js preAction 钩子

这是启动流程中最关键的部分。Commander.js 的 `preAction` 钩子在命令执行前触发，但**不在 `--help` 时触发**：

```typescript
program.hook('preAction', async (thisCommand, actionCommand) => {
  // 1. 等待MDM和Keychain完成
  await Promise.all([mdmReadPromise, keychainPromise]);

  // 2. 核心初始化
  await init();

  // 3. 设置进程标题
  process.title = 'claude-code';

  // 4. 初始化分析sink
  await initSinks();

  // 5. 插件目录设置
  await setupPluginDirectory();

  // 6. 运行配置迁移
  await runMigrations();

  // 7. 远程管理设置（fire-and-forget）
  loadRemoteManagedSettings().catch(() => {}); // 静默失败

  // 8. 加载策略限制
  await loadPolicyLimits();

  // 9. 上传用户设置（遥测）
  uploadUserSettings(); // fire-and-forget
});
```

### 为什么用 preAction 而非直接在 main() 中初始化

这是一个精妙的设计决策：

1. **避免 `--help` 触发副作用**：`claude --help` 不应该初始化 Keychain、连接 API 或加载远程设置。preAction 钩子只在实际执行命令时触发。
2. **避免环境变量信号传递**：如果用环境变量来判断是否初始化，会引入隐式依赖。preAction 是显式的命令生命周期钩子。
3. **保证执行顺序**：preAction 内的 `await` 保证所有初始化按序完成后才执行实际命令。

### fire-and-forget 的远程设置

`loadRemoteManagedSettings()` 使用 fire-and-forget 模式（`.catch(() => {})`），原因：
- 远程设置加载依赖网络，可能耗时几百毫秒到几秒
- 这是一个 fail-open 设计：如果远程设置服务不可用，应用应使用本地设置正常启动
- 企业场景下，远程设置提供额外的策略覆盖，但不应成为启动的阻塞点

## 7. 执行模式路由

初始化完成后，系统根据参数和环境将执行路由到6种模式之一：

### 7.1 Interactive/REPL 模式（6条子路径）

```
Interactive Mode
├── continue (--continue/-c)     → 加载最近会话，恢复上下文
├── direct-connect               → 通过DirectConnect API连接
├── SSH                          → SSH远程模式
├── assistant                    → KAIROS助手模式
├── resume (--resume/-r)         → 恢复指定session ID
└── new (default)                → 全新会话
```

每条子路径的初始化不同：
- **continue**：从 `~/.claude/sessions/` 读取最近会话的消息历史
- **direct-connect**：建立 WebSocket 连接到远端，获取 session_id
- **SSH**：通过 SSH 隧道建立连接，转发终端 I/O
- **assistant**：加载 KAIROS 特定的系统提示和工具集
- **resume**：按 session ID 精确恢复，包括工具状态
- **new**：创建新 session ID，初始化空消息列表

### 7.2 Headless/Print 模式

```typescript
if (hasPrintFlag) {
  const result = await runPrintMode({
    prompt: args.join(' '),
    outputFormat: options.outputFormat,
    maxTurns: options.maxTurns,
  });
  process.stdout.write(formatOutput(result, options.outputFormat));
  process.exit(0);
}
```

Print 模式不渲染 Ink UI，直接将结果写入 stdout。支持三种输出格式：
- `text`：纯文本，适合管道
- `json`：完整 JSON 对象，适合程序化消费
- `stream-json`：NDJSON 流，适合实时处理

### 7.3 Init-Only 模式

仅执行初始化（创建 `CLAUDE.md`、设置配置），然后退出。用于 CI/CD 环境的项目初始化。

### 7.4 SDK/Bridge 模式

通过 WebSocket 或 SSE 连接到外部编排器（IDE 或 SDK），接收指令并返回结果。

## 8. 延迟预取（Deferred Prefetches）

在第一次 REPL 渲染完成后（用户已经看到提示符），系统在后台启动一系列预取操作：

```typescript
// 在首次渲染后的 requestIdleCallback 中
async function runDeferredPrefetches() {
  // 用户身份初始化
  initUser();

  // 用户上下文（偏好、历史）
  getUserContext();

  // 系统上下文（git状态、项目结构）
  prefetchSystemContext();

  // 相关提示建议
  getRelevantTips();

  // 云凭证预取
  prefetchAWSCredentials();
  prefetchGCPCredentials();

  // 文件计数（用于上下文估算）
  countFilesRoundedRg();

  // 分析门控初始化
  initializeAnalyticsGates();

  // 设置变更检测器
  settingsChangeDetector();

  // Skill变更检测器
  skillChangeDetector();
}
```

### 为什么延迟到首次渲染后

- **避免阻塞首次绘制**：用户看到提示符（first paint）是感知启动速度的关键指标
- **利用空闲时间**：用户开始输入前通常有 1-3 秒的停顿，这段时间可以预取
- **容错设计**：这些预取全部是 fire-and-forget，任何一个失败都不影响核心功能
- **按需加载**：如果用户的第一个请求在预取完成前到来，系统会等待相关预取完成

## 9. 启动时序图

```
时间 ──────────────────────────────────────────────────────────►

模块级（0ms）
├─ profileCheckpoint                     ●
├─ startMdmRawRead ─────────────[spawn plutil]──────────────● (~135ms)
├─ startKeychainPrefetch ──────[keychain read]────● (~65ms)
│
import 链求值（~5ms → ~140ms）
├─ TypeScript 模块解析 ──────────────────────────●
│
main() (~140ms)
├─ SIGINT handler 注册                   ●
├─ argv 快速解析                          ●
├─ 交互模式检测                           ●
│
Commander preAction (~150ms)
├─ await MDM + Keychain ─────────────────● (已完成，近乎0ms)
├─ init() ──────────[auth, config]───────● (~200ms)
├─ initSinks() ──────────────────────────● (~10ms)
├─ setupPluginDirectory ─────────────────● (~5ms)
├─ runMigrations ────────────────────────● (~20ms)
├─ loadRemoteManagedSettings ────────────→ (fire-and-forget)
├─ loadPolicyLimits ─────────────────────● (~30ms)
│
执行模式路由 (~420ms)
├─ REPL渲染 ─────────────────────────────● 首次绘制 (~450ms)
│
延迟预取 (~450ms+)
├─ initUser ─────────────────────────────→
├─ getUserContext ───────────────────────→
├─ prefetchSystemContext ────────────────→
├─ getRelevantTips ──────────────────────→
├─ AWS/GCP credentials ─────────────────→
└─ countFilesRoundedRg ─────────────────→
```

关键观察：
1. MDM 读取和 Keychain 预取与模块导入**完全并行**
2. 到 `preAction` 时，MDM 和 Keychain 的 Promise 通常已经 resolved，`await` 几乎不等待
3. 首次绘制目标 ~450ms，延迟预取在此之后
4. 远程设置和用户设置上传是 fire-and-forget，不阻塞任何路径

## 10. 小结

Claude Code 的启动流程体现了几个核心工程理念：

1. **并行一切可并行的**：模块级副作用利用模块求值时间窗口并行 I/O
2. **fail-open 设计**：远程设置、分析、预取的失败不影响核心功能
3. **首次绘制优先**：所有非关键初始化延迟到用户可见后
4. **显式生命周期**：preAction 钩子明确界定了"设置阶段"和"执行阶段"
5. **双模式适配**：SIGINT、输出格式、UI 渲染都根据交互/非交互模式分叉
