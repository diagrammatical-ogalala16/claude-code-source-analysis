## 2. buildTool 工厂函数

`buildTool` 是创建工具实例的标准方式，提供了一组安全的默认值：

```typescript
function buildTool<Input, Output, Progress = never>(
  definition: PartialTool<Input, Output, Progress>
): Tool<Input, Output, Progress> {
  return {
    // === 安全默认值 (TOOL_DEFAULTS) ===
    isEnabled: () => true,
    isConcurrencySafe: false,    // 默认不安全 → 串行执行
    isReadOnly: false,            // 默认非只读 → 需要权限检查
    isDestructive: false,         // 默认非破坏性
    checkPermissions: () => ({ decision: 'allow' }),
    toAutoClassifierInput: () => '',
    userFacingName: definition.name,

    // === 用户定义覆盖 ===
    ...definition,
  };
}
```

### 为什么默认值选择 fail-closed

- `isConcurrencySafe: false`：假设写入直到被证明安全。一个未正确标记的并发执行可能导致数据竞争。
- `isReadOnly: false`：假设有副作用直到被证明只读。错误标记为只读的写操作会跳过权限检查。
- `isDestructive: false`：这个默认看似与 fail-closed 矛盾，但实际上 `isDestructive` 只影响 UI 提示的严重程度，不影响安全决策。

### 类型级展开（Type-Level Spread）

```typescript
type PartialTool<I, O, P> = Partial<Tool<I, O, P>> & Required<Pick<Tool<I, O, P>,
  'name' | 'description' | 'inputSchema' | 'call'
>>;
```

只有 `name`、`description`、`inputSchema`、`call` 是必须提供的。其余 26+ 字段都有合理的默认值。这降低了创建新工具的门槛——最简单的工具只需4个字段。

## 3. ToolUseContext

`ToolUseContext` 是传递给每个工具执行的上下文对象：

```typescript
interface ToolUseContext {
  // 可用资源
  options: {
    commands: Command[];           // 可用命令列表
    tools: Tool[];                 // 可用工具池
    mcpClients: MCPClient[];       // MCP客户端连接
    mcpResources: MCPResource[];   // MCP资源
    mainLoopModel: string;         // 主循环使用的模型
    thinkingConfig: ThinkingConfig; // 思考配置
    agentDefinitions: AgentDef[];  // Agent定义列表
    maxBudgetUsd: number;          // 最大预算
    customSystemPrompt?: string;   // 自定义系统提示
    isNonInteractiveSession: boolean; // 是否非交互
    refreshTools: () => void;      // 工具池刷新函数
  };

  // 控制流
  abortController: AbortController;

  // 文件系统状态
  readFileState: ReadFileState;

  // 应用状态访问
  getAppState: () => AppState;
  setAppState: (updater: (s: AppState) => AppState) => void;

  // 对话框
  handleElicitation: (req: ElicitationRequest) => Promise<ElicitationResponse>;

  // UI注入
  setToolJSX: (jsx: React.ReactNode) => void;
}
```

### 为什么不直接传递 Store

ToolUseContext 不直接暴露 Store 对象，而是通过 `getAppState/setAppState` 间接访问。原因：

1. **控制表面积**：工具只能通过规定的接口修改状态，不能直接 subscribe 或重置
2. **可测试性**：测试中可以提供 mock 的 get/set 函数
3. **审计性**：所有状态修改通过同一接口，便于日志记录

### 4.3 assembleToolPool() 排序策略

```typescript
function assembleToolPool(builtIn: Tool[], mcpTools: Tool[]): Tool[] {
  // 内置工具排序：按固定顺序
  const sortedBuiltIn = builtIn.sort((a, b) =>
    TOOL_ORDER.indexOf(a.name) - TOOL_ORDER.indexOf(b.name)
  );

  // MCP工具排序：按服务器名+工具名
  const sortedMCP = mcpTools.sort((a, b) =>
    `${a.serverName}:${a.name}`.localeCompare(`${b.serverName}:${b.name}`)
  );

  // 内置在前，MCP在后
  return [...sortedBuiltIn, ...sortedMCP];
}
```

### 为什么分开排序

**Prompt 缓存命中率**。Anthropic 的 prompt caching 是前缀敏感的——系统提示中工具定义的顺序变化会导致缓存失效。通过将内置工具和 MCP 工具分别固定排序：

- 添加/删除 MCP 服务器不会改变内置工具的顺序（保护前缀缓存）
- 同一 MCP 服务器的工具更新不会改变其他服务器的工具顺序
- 内置工具在前意味着大部分缓存前缀是稳定的

## 5. 并发分区执行

### 5.1 分区逻辑

```typescript
function partitionToolCalls(calls: ToolCall[]): ToolCallGroup[] {
  // 核心逻辑：连续的并发安全只读调用可以批量执行
  // 非安全或非只读的调用必须单独串行执行
}
```

规则：
- `isConcurrencySafe === true` AND `isReadOnly === true` → 可以并发
- 其他任何组合 → 必须串行
- 连续的可并发调用被归入同一批次

### 5.2 并发执行限制

```typescript
async function runToolsConcurrently(
  calls: ToolCall[],
  context: ToolUseContext,
  maxConcurrency: number = 10
): Promise<ToolResult[]> {
  const semaphore = new Semaphore(maxConcurrency);
  return Promise.all(
    calls.map(async (call) => {
      await semaphore.acquire();
      try {
        return await executeTool(call, context);
      } finally {
        semaphore.release();
      }
    })
  );
}
```

最大并发数为 10，原因：
- 防止文件描述符耗尽（每个工具可能打开文件或 spawn 进程）
- 防止系统负载过高（ripgrep、LSP 等外部进程）
- 10 是经验值：覆盖了模型一次性发起的典型只读操作数量（5-8个）

### 5.3 Context Modifier 队列

```typescript
// 工具执行返回context modifier
interface ToolResult {
  output: any;
  contextModifier?: (messages: Message[]) => Message[];
}

// 按toolUseID顺序应用
for (const id of Object.keys(queuedContextModifiers).sort()) {
  state.messages = queuedContextModifiers[id](state.messages);
}
```

Context modifier 在所有工具执行完成后才应用，且按 ID 顺序串行应用。这保证了：
- 并发执行的工具不会相互干扰消息状态
- modifier 的应用顺序是确定的（按工具调用顺序）

## 6. 流式工具执行（StreamingToolExecutor）

当模型正在流式输出多个工具调用时，StreamingToolExecutor 可以在接收完一个工具调用的参数后立即开始执行，无需等待所有工具调用接收完毕。

### 6.1 TrackedTool 状态

```typescript
interface TrackedTool {
  toolCall: ToolCall;
  state: 'queued' | 'executing' | 'completed' | 'yielded';
  result?: ToolResult;
  error?: Error;
}
```

状态转换：
```
queued → executing → completed → yielded
                  ↘ error → yielded
```

### 6.2 队列处理条件

```typescript
function processQueue() {
  for (const tracked of queue) {
    if (tracked.state === 'queued' && canExecute(tracked)) {
      tracked.state = 'executing';
      execute(tracked);
    }
  }
}

function canExecute(tracked: TrackedTool): boolean {
  // 只有当前面所有非并发安全的工具都已完成时才能执行
  const index = queue.indexOf(tracked);
  for (let i = 0; i < index; i++) {
    if (!queue[i].toolCall.tool.isConcurrencySafe && queue[i].state !== 'completed') {
      return false;
    }
  }
  return true;
}
```

### 6.3 兄弟中止（Sibling Abort）

```typescript
// 如果一个工具执行出错，中止其他正在执行的兄弟工具
function onToolError(failed: TrackedTool) {
  for (const tracked of queue) {
    if (tracked.state === 'executing' && tracked !== failed) {
      tracked.abortController.abort('sibling_error');
    }
  }
}
```

兄弟中止的原因：一个工具的失败通常意味着整个工具批次的语义失效（例如，第一步搜索失败了，后续的文件读取没有意义）。

## 7. 工具结果大小管理

### 7.1 大小限制

```typescript
const DEFAULT_MAX_RESULT_SIZE_CHARS = 50_000; // 50K字符

function processToolResult(tool: Tool, result: any): ToolResultBlock {
  const maxSize = getMaxResultSize(tool);
  const serialized = serialize(result);

  if (serialized.length > maxSize) {
    // 写入磁盘，返回预览
    const diskPath = writeToDisk(serialized);
    const preview = serialized.slice(0, 2000);
    return {
      type: 'tool_result',
      content: `[Result truncated. Full result saved to ${diskPath}]\n\n${preview}`,
    };
  }

  return { type: 'tool_result', content: serialized };
}
```

### 7.2 GrowthBook 覆盖

```typescript
// 特定工具可以通过GrowthBook配置自定义大小限制
const overrideMap = getFeatureValue('tool_result_size_overrides', {});
function getMaxResultSize(tool: Tool): number {
  return overrideMap[tool.name] ?? tool.maxResultSizeChars ?? DEFAULT_MAX_RESULT_SIZE_CHARS;
}
```

### 为什么写入磁盘

大型工具结果（如完整文件内容、搜索结果）直接放入上下文会：
- 消耗宝贵的 token 预算
- 降低模型对关键信息的注意力
- 可能导致上下文溢出

写入磁盘 + 2KB 预览是一个折中方案：模型看到预览就能判断是否需要更详细的信息，如果需要可以通过文件读取工具获取完整内容。

## 9. 延迟工具加载

### 9.1 shouldDefer 标志

```typescript
const WebFetchTool = buildTool({
  name: 'WebFetch',
  shouldDefer: true,
  searchHint: 'web fetch http url download page',
  // ...
});
```

标记为 `shouldDefer` 的工具不会出现在初始工具列表中（减少系统提示的 token 消耗），而是通过 ToolSearch 工具按需发现和加载。

### 9.2 ToolSearch 发现流程

```
模型需要获取网页 → 调用 ToolSearch("fetch web page")
→ ToolSearch 匹配 WebFetchTool 的 searchHint
→ 返回工具定义
→ 工具被添加到当前会话的可用工具池
→ 模型现在可以使用 WebFetch
```

### 为什么延迟加载

Claude Code 有 39+ 工具，如果全部放入系统提示：
- 消耗约 15K-20K tokens 仅用于工具定义
- 大部分工具在普通会话中从未使用
- prompt cache 失效更频繁（任何工具定义变更都影响缓存）

延迟加载将基础工具集减少到 ~10 个核心工具，需要时通过 ToolSearch 动态扩展。

## 10. 惰性 Schema 加载

```typescript
// 避免循环依赖的惰性schema
const AgentTool = buildTool({
  name: 'Agent',
  lazyInputSchema: () => {
    const { AgentInputSchema } = require('./agent/schemas');
    return AgentInputSchema;
  },
  // ...
});
```

### 为什么需要惰性 Schema

AgentTool 的输入 schema 引用了 `AgentDefinition`，而 `AgentDefinition` 又引用了 `Tool` 类型——形成循环依赖。`lazySchema()` 将 schema 的求值延迟到首次使用时，此时所有模块都已加载完毕。

同时，惰性加载加速了启动：不是所有工具的 Zod schema 都需要在启动时编译（Zod v4 的 schema 编译有非零开销）。

## 11. 设计决策总结

| 设计决策 | 原因 |
|---------|------|
| fail-closed 默认值 | 安全优先于便利 |
| 分离排序内置/MCP工具 | 保护prompt cache前缀 |
| 最大并发=10 | 平衡吞吐与系统资源 |
| 结果写磁盘+预览 | 防止上下文污染 |
| 延迟工具加载 | 减少基础prompt大小 |
| 惰性Schema | 解决循环依赖+加速启动 |
| Context Modifier 队列 | 隔离并发工具的状态修改 |
| 兄弟中止 | 防止无意义的继续执行 |
