## 项目背景

2026年3月31日，Anthropic 的 Claude Code CLI 工具源码通过 npm 包中未清理的 `.map` 文件泄露。这一发现由 Twitter 用户 @Fried_rice 首先公开。`.map` 文件（Source Map）是 JavaScript/TypeScript 构建工具生成的调试映射文件，包含了从编译后代码到原始源码的完整映射，使得完整的 TypeScript 源码得以还原。

本系列文档基于还原后的源码进行深度分析，旨在理解 Claude Code 作为一个工业级 AI 编程助手的完整架构设计、安全机制、性能优化和工程决策。

## 架构哲学

Claude Code 的架构设计遵循以下核心哲学：

### 模块化（Modular）
每个子系统（工具、服务、UI、权限）都有清晰的边界和接口。工具通过 `buildTool` 工厂注册，服务通过依赖注入解耦，UI 通过 React 组件化。这使得单个模块可以独立开发、测试和替换。

### 权限优先（Permission-First）
安全不是事后添加的功能，而是架构的第一公民。每个工具调用都经过权限决策管道，BashTool 有三层独立的安全检查，文件操作有设备路径封锁，远程会话有viewer-only模式。默认策略是 fail-closed：工具默认不支持并发（`isConcurrencySafe=false`），直到被显式标记为安全。

### 多执行模式统一（Multi-Execution-Mode Unity）
六种执行模式共享同一套核心代码：

1. **REPL 模式** - 交互式终端，用户直接使用
2. **Bridge 模式** - IDE集成（VSCode/JetBrains），通过WebSocket/SSE通信
3. **SDK/Print 模式** - 无头执行，结构化输出
4. **Coordinator 模式** - 多Agent协调器，管理工作团队
5. **Assistant (KAIROS) 模式** - Anthropic内部助手模式
6. **Worktree 模式** - Git worktree隔离的并行执行

### 单代码库 + Feature Flag DCE（Single Codebase with Dead Code Elimination）
所有部署变体（外部公开版、Anthropic内部版、Assistant版、Bridge版）从同一源码构建。构建时 Feature Flag 通过 `feature()` 函数实现，Bun 的 bundler 在编译时将未启用的分支完全剔除（Dead Code Elimination），确保外部版本不包含任何内部功能代码。

这种设计既保证了代码一致性（所有变体共享核心逻辑），又保证了安全性（内部功能不会泄露到外部构建中）。
