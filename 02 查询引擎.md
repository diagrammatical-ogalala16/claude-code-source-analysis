# 第2章 查询引擎

> **作者**: [Wang Yanshu](https://catyans.github.io)

> 本章深入分析 Claude Code 的核心组件 QueryEngine.ts 和 query.ts（68KB），涵盖查询循环状态机、API 流式消费、错误恢复级联和 Token 预算管理。

## 1. QueryEngine 类

QueryEngine 是整个系统的心脏，负责管理与 Anthropic API 的完整交互生命周期。

### 1.1 核心属性

```typescript
class QueryEngine {
  // 消息管理
  mutableMessages: Message[];           // 可变消息列表，整个会话的对话历史

  // 控制流
  abortController: AbortController;     // 取消信号，贯穿整个查询链

  // 使用量追踪
  totalUsage: TokenUsage;               // 累计token使用量

  // 文件系统状态
  readFileState: ReadFileState;          // 已读取文件的缓存状态

  // 权限追踪
  permissionDenials: Set<string>;       // 被拒绝的权限记录

  // Skill发现
  discoveredSkillNames: Set<string>;    // 已发现的Skill名称

  // 记忆路径
  loadedNestedMemoryPaths: Set<string>; // 已加载的嵌套Memory文件
}
```

### 为什么用可变消息列表

`mutableMessages` 使用可变数组而非不可变数据结构，因为：
- 对话历史可能很长（数百条消息），每次创建新数组的内存开销不可接受
- 消息修改频繁（compaction、tool result 注入、context modifier 应用）
- 只有 QueryEngine 自身修改此数组，无需并发保护

### ReadFileState 的角色

`readFileState` 缓存了文件读取结果和文件系统元数据（修改时间、大小），使得：
- 同一文件不会被重复读取（即使多个工具请求同一文件）
- 文件变更可以被检测（对比缓存的 mtime）
- 投机执行可以预测文件内容（见 [[16 数据流与架构模式]]）

## 2. 查询循环状态机

query.ts 的核心是一个 AsyncGenerator 状态机。每次循环迭代代表一轮 API 调用和可能的工具执行。

### 2.1 State 类型定义

```typescript
type State = {
  messages: Message[];
  toolUseContext: ToolUseContext;
  autoCompactTracking: {
    consecutiveFailures: number;      // 连续压缩失败次数
    lastAttemptTokenCount: number;    // 上次尝试时的token数
  };
  maxOutputTokensRecoveryCount: number; // 已尝试的输出token恢复次数
  hasAttemptedReactiveCompact: boolean; // 是否已尝试响应式压缩
  maxOutputTokensOverride: number | null; // token上限覆盖值
  pendingToolUseSummary: string | null;   // 待处理的工具使用摘要
  stopHookActive: boolean;               // 停止钩子是否活跃
  turnCount: number;                     // 当前轮次计数
  transition: Transition;                // 状态转换类型
};
```

### 2.2 状态转换（Transition）枚举

```typescript
type Transition =
  | 'continue'           // 正常继续下一轮
  | 'tool_use'           // 有工具需要执行
  | 'stop_sequence'      // 模型停止序列触发
  | 'end_turn'           // 模型主动结束
  | 'max_tokens'         // 达到token上限
  | 'error_recovery'     // 进入错误恢复
  | 'budget_exceeded'    // 预算超限
  | 'abort';             // 用户取消
```

### 2.3 循环主体

```typescript
async function* query(state: State): AsyncGenerator<QueryEvent> {
  while (true) {
    // === Phase 1: Pre-API ===
    yield* preApiPhase(state);

    // === Phase 2: API Request ===
    const response = yield* apiRequestPhase(state);

    // === Phase 3: Response Processing ===
    const transition = yield* processResponse(response, state);

    // === Phase 4: Tool Execution ===
    if (transition === 'tool_use') {
      yield* executeTools(state);
    }

    // === Phase 5: Stop Hooks ===
    if (transition === 'end_turn' || transition === 'stop_sequence') {
      yield* runStopHooks(state);
    }

    // === Phase 6: Transition ===
    if (shouldExit(transition)) break;
    state.turnCount++;
  }
}
```

### 为什么使用 AsyncGenerator

AsyncGenerator（`async function*`）相比 Promise 链或回调有几个关键优势：

1. **背压（Backpressure）**：消费者控制拉取速度，如果 UI 渲染跟不上，生成器自然暂停
2. **惰性求值**：只在 `next()` 被调用时才产出下一个值，不会预计算
3. **取消机制**：`return()` 或 `throw()` 可以在任何 `yield` 点取消执行
4. **渐进式渲染**：每个 `yield` 产出一个事件，UI 可以逐步更新
5. **状态封装**：生成器函数的局部变量就是状态，无需外部状态管理

## 3. Pre-API 阶段

### 3.1 Skill 发现预取

在 API 调用前，系统检查当前对话上下文中是否有需要加载的 Skill：

```typescript
// 并行预取：不阻塞API调用
const skillDiscoveryPromise = discoverSkills(state.messages);
```

### 3.2 内容替换预算（Content Replacement Budget）

对已有的消息进行token预算管理，确保不超过上下文窗口限制。这涉及几种压缩策略的级联调用。

### 3.3 五种压缩策略

1. **Snip Compaction（片段压缩）**：将历史消息中较长的工具输出替换为摘要（"此结果已被省略"）

2. **Micro Compaction（微压缩）**：对单条消息内部的冗余内容进行压缩，例如重复的文件路径或代码块

3. **Context Collapse（上下文折叠）**：将多轮相似的工具调用合并为一个摘要（如"连续读取了5个文件"）

4. **Auto Compaction（自动压缩）**：当 token 使用接近阈值（90%）时触发的完整压缩流程
   - 有**熔断器（Circuit Breaker）**保护：连续失败超过阈值后停止尝试
   - 追踪 `consecutiveFailures` 防止无限重试

5. **Session Memory Compaction（会话记忆压缩）**：去重已经存在于 CLAUDE.md 中的信息

### 为什么需要5种策略

每种策略针对不同的场景：
- Snip：快速释放空间，代价是丢失详细信息
- Micro：精细优化，保留语义
- Context Collapse：减少重复模式的冗余
- Auto：系统级别的完整压缩
- Session Memory：与持久化记忆的去重

## 4. API 请求阶段

### 4.1 阻塞限制检查

```typescript
// 在发起API请求前检查各种限制
await checkBlockingLimits(state);
```

检查项包括：
- Token 预算是否已耗尽
- 美元预算是否已超限
- 轮次限制是否已达到
- 速率限制是否需要等待

### 4.2 withRetry 包装器

```typescript
const response = await withRetry(
  () => makeApiRequest(state),
  {
    maxRetries: MAX_RETRIES,
    baseDelay: 500,          // 基础延迟 500ms
    maxDelay: 32000,         // 最大延迟 32s
    jitter: 0.25,            // ±25% 抖动
    retryableErrors: [429, 529, 'ECONNRESET', 'ETIMEDOUT'],
  }
);
```

重试策略的细节：

```
延迟计算: delay = min(baseDelay * 2^(attempt-1), maxDelay) * (1 ± jitter)

attempt 1: 500ms  * (0.75 ~ 1.25) = 375ms  ~ 625ms
attempt 2: 1000ms * (0.75 ~ 1.25) = 750ms  ~ 1250ms
attempt 3: 2000ms * (0.75 ~ 1.25) = 1500ms ~ 2500ms
attempt 4: 4000ms * (0.75 ~ 1.25) = 3000ms ~ 5000ms
...
attempt N: 32000ms * (0.75 ~ 1.25) = 24000ms ~ 40000ms (capped)
```

特殊重试逻辑：
- **快速模式回退**：429/529 错误时尝试使用更小的模型
- **模型回退**：连续 529 超过 `MAX_529_RETRIES` 后切换到备用模型
- **持久重试**：瞬态容量错误（529）使用无限重试（带心跳保活）
- **心跳保活**：长时间等待期间周期性发送心跳事件，防止上层超时

### 4.3 流式消费

Claude Code 不使用 SDK 的 `BetaMessageStream` 助手类，而是直接消费原始的 SSE 流：

```typescript
// 直接消费原始流
const stream = await client.messages.create({
  ...params,
  stream: true,
});

for await (const event of stream) {
  switch (event.type) {
    case 'message_start':
      handleMessageStart(event);
      break;
    case 'content_block_start':
      handleBlockStart(event);
      break;
    case 'content_block_delta':
      handleDelta(event);
      break;
    case 'content_block_stop':
      handleBlockStop(event);
      break;
    case 'message_delta':
      handleMessageDelta(event);
      break;
    case 'message_stop':
      handleMessageStop(event);
      break;
  }
}
```

### 为什么不用 BetaMessageStream

**避免 O(n^2) 的部分 JSON 解析**。`BetaMessageStream` 在每个 `input_json_delta` 事件到达时，会重新解析已累积的 JSON 字符串（`partialParse()`），这是 O(n) 操作，在 n 个 delta 上变成 O(n^2)。对于大型工具调用（如长代码块），这导致显著的性能下降。

Claude Code 的方案是手动累积 `content_block_delta` 事件中的文本片段，仅在 `content_block_stop` 时一次性解析完整 JSON，保持 O(n) 的总复杂度。

## 5. 空闲看门狗（Idle Watchdog）

### 5.1 主动超时

```typescript
const watchdog = createWatchdog({
  warningTimeout: 45_000,   // 45秒：发出警告
  killTimeout: 90_000,       // 90秒：终止请求
  onWarning: () => yield { type: 'idle_warning' },
  onKill: () => abortController.abort('idle_timeout'),
});
```

看门狗在每次收到流事件时被"喂食"（reset），如果连续 45 秒没有新事件，发出警告；90 秒后强制终止。

### 5.2 被动停滞检测

```typescript
let lastChunkTime = Date.now();

for await (const chunk of stream) {
  const now = Date.now();
  const gap = now - lastChunkTime;

  if (gap > 30_000) {
    yield { type: 'stall_detected', gap };
  }

  lastChunkTime = now;
  // 处理chunk...
}
```

停滞检测是被动的——只有在收到下一个 chunk 时才检查间隔。如果间隔超过 30 秒，标记为停滞。

### 为什么需要两种机制

- **看门狗（主动）**：处理完全断连场景（服务器崩溃、网络断开），独立计时器自行触发
- **停滞检测（被动）**：处理慢速场景（服务器过载但未断连），依赖下一个事件触发

两者互补：看门狗确保最终超时，停滞检测提供更细粒度的延迟信息用于 UI 反馈。

## 6. 扣留机制（Withholding Mechanism）

```typescript
// 可恢复错误被扣留，直到恢复尝试完成
const withheldEvents: QueryEvent[] = [];

function yield_or_withhold(event: QueryEvent) {
  if (isRecoverableError(event) && !allRecoveryAttempted()) {
    withheldEvents.push(event);
  } else {
    yield event;
    // 恢复成功后，扣留的事件被丢弃
    // 恢复失败后，扣留的事件被全部yield
  }
}
```

### 为什么需要扣留

当 API 返回错误（如 `max_tokens` 被截断），系统会尝试自动恢复（如扩大输出限制后重试）。在恢复期间，不应该把错误事件传给 UI，否则用户会看到一个即将被修复的错误闪烁。

只有当所有恢复尝试都失败后，扣留的错误事件才被释放给 UI 显示。

## 7. 七级错误恢复级联

当 API 请求或响应处理出错时，系统按优先级尝试7种恢复策略：

### Level 1: Streaming Fallback（流式回退）

如果流式连接失败（如代理不支持 SSE），回退到非流式请求。

```typescript
if (streamingFailed && !hasTriedNonStreaming) {
  response = await makeNonStreamingRequest(params);
}
```

### Level 2: Collapse Drain（折叠清空）

如果上下文过大导致请求失败，执行紧急上下文折叠，丢弃最旧的消息。

### Level 3: Reactive Compact（响应式压缩）

比 Collapse Drain 更智能的压缩：使用 LLM 对历史消息进行有选择的摘要。

```typescript
if (!state.hasAttemptedReactiveCompact) {
  state.hasAttemptedReactiveCompact = true;
  await reactiveCompact(state.messages);
  continue; // 重试API调用
}
```

注意 `hasAttemptedReactiveCompact` 标志：每个查询循环只尝试一次，防止反复压缩。

### Level 4: Max Output Tokens Escalation（输出Token上限升级）

如果响应因 `max_tokens` 被截断，将输出限制提升到 64K。

```typescript
if (stopReason === 'max_tokens' && state.maxOutputTokensRecoveryCount === 0) {
  state.maxOutputTokensOverride = 64_000;
  state.maxOutputTokensRecoveryCount++;
  continue;
}
```

### Level 5: Max Output Tokens Multi-Turn（多轮Token续接）

如果单次扩大仍不够，允许最多3次续接请求。

```typescript
if (stopReason === 'max_tokens' && state.maxOutputTokensRecoveryCount < 3) {
  state.messages.push(assistantMessage);
  state.messages.push({ role: 'user', content: 'Please continue.' });
  state.maxOutputTokensRecoveryCount++;
  continue;
}
```

### Level 6: Stop Hook Blocking（停止钩子阻塞）

如果停止钩子（如自定义验证逻辑）返回 "block"，重新进入查询循环。

### Level 7: Token Budget Continuation（Token预算续接）

当 BudgetTracker 检测到模型正在做有用工作但接近预算限制时，允许额外续接。

### 级联设计原则

- **每种恢复只尝试一次**：通过各自的守卫标志防止重入（`hasAttemptedReactiveCompact`、`maxOutputTokensRecoveryCount`）
- **从轻量到重量**：先尝试低成本恢复（流式回退），再尝试高成本恢复（LLM 压缩）
- **无交叉影响**：每级恢复独立运行，不依赖其他级别的状态

## 8. withRetry 包装器详解

```typescript
async function withRetry<T>(
  fn: () => Promise<T>,
  options: RetryOptions,
): Promise<T> {
  let attempt = 0;

  while (true) {
    try {
      return await fn();
    } catch (error) {
      attempt++;

      if (!isRetryable(error, options.retryableErrors)) {
        throw error;
      }

      // 快速模式回退
      if (is429or529(error) && canFallbackToFastModel()) {
        switchToFastModel();
        continue;
      }

      // 模型回退
      if (is529(error) && attempt > MAX_529_RETRIES) {
        switchToFallbackModel();
      }

      // 计算延迟
      const delay = Math.min(
        options.baseDelay * Math.pow(2, attempt - 1),
        options.maxDelay
      );
      const jitteredDelay = delay * (1 + (Math.random() - 0.5) * 2 * options.jitter);

      // 心跳保活
      const heartbeatInterval = setInterval(() => {
        emitHeartbeat();
      }, 10_000);

      await sleep(jitteredDelay);
      clearInterval(heartbeatInterval);
    }
  }
}
```

### 心跳保活的必要性

在长时间重试等待（最长 32 秒）期间，上层可能认为进程卡死。心跳事件每 10 秒发出一次，让 UI 可以显示"正在重试..."的状态，也防止 Bridge/SDK 连接超时断开。

## 9. Token 预算管理

### BudgetTracker

```typescript
class BudgetTracker {
  continuationCount: number = 0;

  shouldContinue(usage: TokenUsage, isSubagent: boolean): boolean {
    // 子代理总是停止——不参与预算管理
    if (isSubagent) return false;

    // 检查预算阈值（90%）
    if (usage.totalTokens > budget * 0.9) return false;

    // 检查递减收益（3+次续接且每次<500 token）
    if (this.continuationCount >= 3 && lastOutputTokens < 500) return false;

    this.continuationCount++;
    return true;
  }
}
```

### 为什么子代理不参与预算

子代理（subagent）立即停止的原因：
- 预算会计复杂性：跨多个代理追踪和分配预算需要复杂的同步机制
- 避免预算争抢：主线程和子代理可能同时消费预算，导致超支
- 简化模型：子代理完成单一任务后返回，主线程负责全局预算

### 递减收益检测

3次以上续接且每次产出少于 500 token 表明模型正在"磨蹭"——反复输出无意义的内容来填充 token。此时停止可以节省预算。

## 10. 停止钩子执行顺序

当模型决定停止（`end_turn` 或 `stop_sequence`）时，按以下顺序执行钩子：

```
1. 缓存安全参数（cache safe params）
2. 模板任务分类（template job classification）
3. 背景记录任务（fire-and-forget）
   ├── 提示建议生成（prompt suggestion）
   ├── 记忆提取（memory extraction）
   └── 自动梦境（auto-dream）
4. Computer Use 清理
5. 停止钩子执行（stop hooks execution）
6. 队友钩子
   ├── TaskCompleted
   └── TeammateIdle
```

### 为什么背景记录任务是 fire-and-forget

- 这些任务不影响用户可见的输出
- 不应阻塞轮次完成（用户等待时间）
- 失败是可接受的（记忆提取失败只是丢失一次学习机会）

## 11. 工具执行调度

### 分区策略

```typescript
function partitionToolCalls(toolCalls: ToolCall[]): ToolCallGroup[] {
  const groups: ToolCallGroup[] = [];
  let currentBatch: ToolCall[] = [];

  for (const call of toolCalls) {
    if (call.tool.isConcurrencySafe && call.tool.isReadOnly) {
      currentBatch.push(call);
    } else {
      if (currentBatch.length > 0) {
        groups.push({ type: 'concurrent', calls: currentBatch });
        currentBatch = [];
      }
      groups.push({ type: 'serial', calls: [call] });
    }
  }

  if (currentBatch.length > 0) {
    groups.push({ type: 'concurrent', calls: currentBatch });
  }

  return groups;
}
```

### 执行流程

1. **分区（Partition）**：将工具调用分为并发安全组和串行组
2. **并发执行**：安全组最多 10 个并发（`Promise.all` 带限制）
3. **串行执行**：非安全组逐个执行
4. **Context Modifier 应用**：工具执行完成后，按 toolUseID 顺序应用 Context Modifier

## 12. 附件收集

每轮 API 调用前，系统收集多种"附件"注入到上下文中：

```typescript
function collectAttachments(state: State): Attachment[] {
  return [
    ...getQueuedCommands(state),         // 排队的用户命令
    ...getFileChanges(state),            // 文件变更通知
    ...getSkillAttachments(state),       // Skill上下文
    ...consumeMemoryPrefetch(state),     // 记忆预取结果
  ];
}
```

### 为什么分散收集而非集中

各类附件的生产者分散在不同子系统中（命令系统、文件监控、Skill系统、记忆系统）。收集阶段统一拉取，避免了各子系统需要了解 API 请求格式的耦合。

## 13. 设计总结

查询引擎的核心设计决策：

| 决策 | 原因 |
|------|------|
| AsyncGenerator 状态机 | 背压、取消、渐进式渲染 |
| 原始流消费（非 BetaMessageStream） | 避免 O(n^2) 的部分 JSON 解析 |
| 扣留机制 | 防止用户看到即将被修复的错误 |
| 7级恢复级联 | 每级独立、单次尝试、从轻到重 |
| 子代理不参与预算 | 简化跨代理预算会计 |
| Fire-and-forget 背景任务 | 不阻塞轮次完成 |
| 工具分区执行 | 读安全并发、写必须串行 |
