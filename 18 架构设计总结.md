# 第18章 架构设计总结

> **作者**: [Wang Yanshu](https://catyans.github.io)

> 本章是对 Claude Code 整体架构的综合总结，涵盖核心设计原则、代码分布、关键模式、性能优化、可扩展性模型和安全架构。

## 1. 七大核心设计原则

### 1.1 安全优先（Security First）

安全不是后加的功能，而是渗透在每一层的基本约束：

- **BashTool 三层安全**：23 项语法检查 → 权限规则匹配 → 只读模式强制（见 [[04 工具详解-BashTool]]）
- **ResolveOnce 竞态防护**：双标志（claimed + delivered）确保权限决策原子性（见 [[07 权限系统]]）
- **fail-closed 工具默认**：`isConcurrencySafe=false`，`isReadOnly=false`——假设最坏情况直到被证明安全（见 [[03 工具系统]]）
- **O_EXCL + O_NOFOLLOW**：Skill 文件提取的 TOCTOU 和符号链接防护（见 [[12 Skill与插件系统]]）
- **设备路径封锁**：阻止读取 `/dev/zero`、`/proc/self/environ` 等危险路径（见 [[06 工具详解-文件与搜索工具]]）
- **Token 泄露防护**：团队记忆文件中的密钥检测（见 [[11 记忆系统]]）

**设计哲学**：在安全和便利之间，Claude Code 选择了安全。用户可以通过配置显式放宽限制（`--permission-mode auto`），但默认行为是保守的。

### 1.2 构建时优化（Build-Time Optimization）

单代码库服务四种部署变体，通过编译时 DCE 实现：

```
                    ┌─ External（公开）
                    │  大部分 feature = false
                    │
源代码 ─── DCE ────├─ Ant-Internal（内部）
                    │  所有 feature = true
                    │
                    ├─ Assistant（KAIROS）
                    │  选择性 feature
                    │
                    └─ Bridge（IDE集成）
                       BRIDGE_MODE = true
```

**好处**：
- 外部构建中**物理上不存在**内部功能代码（不仅是运行时禁用）
- Bug 修复自动传播到所有变体
- 一套 CI/CD 管线覆盖所有变体

### 1.3 启动性能（Startup Performance）

```
目标：用户看到提示符 < 500ms

优化手段：
├─ 模块级并行预取（~65ms savings）
├─ 惰性工具加载（减少初始工具集）
├─ 惰性 Schema 编译（按需 Zod 编译）
├─ 延迟预取（首次渲染后）no
├─ Fire-and-forget 远程设置
└─ preAction 守卫（--help 不初始化）
```

详见 [[01 启动流程]]。

### 1.4 全面流式化（Streaming Everything）

AsyncGenerator 是系统的统一流式原语：

```
查询循环   ─── AsyncGenerator<QueryEvent>
API 流     ─── AsyncGenerator<StreamEvent>
工具执行   ─── AsyncGenerator<ToolProgress, ToolResult>
Hook 执行  ─── AsyncGenerator<HookEvent>
```

这带来的统一好处：
- **取消**：`generator.return()` 在任何 `yield` 点终止
- **背压**：消费者控制拉取速度
- **组合**：`yield*` 委托给子生成器
- **渐进渲染**：每个 yield 触发 UI 更新

### 1.5 不可变状态（Immutable State）

`DeepImmutable<AppState>` 确保 150+ 字段的状态树不被意外修改：

```typescript
// 编译错误：不能直接修改
state.verbose = true;           // ❌ readonly

// 正确：通过setState创建新状态
store.setState(s => ({
  ...s,
  verbose: true,               // ✅ 新对象
}));
```

详见 [[08 状态管理]]。

### 1.6 渐进降级（Progressive Degradation）

七级错误恢复级联确保系统在面对各种故障时尽可能继续工作：

```
Level 1: 流式 → 非流式回退
Level 2: 上下文折叠
Level 3: 响应式压缩
Level 4: 输出 Token 上限升级
Level 5: 多轮续接
Level 6: 停止钩子阻塞
Level 7: Token 预算续接
```

每级独立运行，单次尝试，从轻到重。详见 [[02 查询引擎]]。

### 1.7 多模式统一（Multi-Mode Unity）

六种执行模式共享一个核心：

| 模式 | 入口 | UI | 权限 |
|------|------|-----|------|
| REPL | CLI | Ink React | Interactive |
| Bridge | IDE | 无（IDE渲染） | Bridge CCR |
| SDK/Print | API | stdout | 程序化 |
| Coordinator | CLI | Ink React | Coordinator |
| Assistant | CLI | Ink React | Custom |
| Worktree | CLI | Ink React | Interactive |

共享组件：QueryEngine、工具系统、权限管道、消息规范化、上下文压缩。

## 3. 关键架构模式

| 模式 | 描述 | 使用位置 | 为什么 |
|------|------|---------|--------|
| AsyncGenerator 状态机 | 生成器驱动的状态转换 | 查询循环、工具执行 | 取消、背压、渐进渲染 |
| ResolveOnce | 多竞争者的原子决策 | 权限系统 | 防止竞态 |
| Context Modifier | 延迟的上下文修改队列 | 工具执行后 | 并发安全、确定性顺序 |
| 双缓冲渲染 | front/back 屏幕缓冲交换 | UI 渲染 | 防闪烁、最小输出 |
| 投机执行 | 流式预测文件写入 | 文件操作 | 减少感知延迟 |
| Fire-and-forget | 异步不等待 | 分析、记忆提取 | 不阻塞用户 |
| Feature Flag DCE | 编译时代码消除 | 构建系统 | 安全、体积 |
| 惰性加载 | 首次使用时加载 | 工具、Schema、Skill | 加速启动 |
| 熔断器 | 连续失败后停止 | 自动压缩 | 防止无效重试 |
| 分层安全 | 多独立安全检查层 | BashTool | 纵深防御 |
| 三路权限处理 | Interactive/Worker/Coordinator | 权限系统 | 适配不同执行上下文 |
| 依赖注入 | 可替换的外部依赖 | Bridge | 可测试性 |

## 4. 性能优化全景

### 4.1 启动优化（~65ms 节省）

| 优化 | 节省 | 机制 |
|------|------|------|
| Keychain 并行预取 | ~65ms | 模块级 spawn 并行 |
| MDM 重叠读取 | ~135ms | 与 import 并行 |
| 惰性工具加载 | ~30ms | 减少初始工具数 |
| preAction 守卫 | ~200ms | --help 不初始化 |
| 延迟预取 | 不阻塞首帧 | 首次渲染后执行 |

### 4.2 运行时优化

| 优化 | 效果 | 机制 |
|------|------|------|
| 上下文缓存 | git status 0ms vs 100ms | 按会话缓存 |
| 工具并发 | 10x 读取吞吐 | 只读工具并行 |
| 投机渲染 | ~100ms 感知优化 | 流式预测写入 |
| Prompt 缓存 | ~50% token 节省 | 稳定工具排序 |
| 内容替换 | 按需压缩 | 5种策略级联 |
| Token 预算 | 避免无效续接 | 递减收益检测 |
| 自动压缩 | 90% 阈值触发 | 带熔断器 |

### 4.3 渲染优化

| 优化 | 效果 | 机制 |
|------|------|------|
| 双缓冲 | 无闪烁 | 帧差分输出 |
| 硬件滚动 | O(1) vs O(n) 滚动 | DECSTBM |
| 虚拟滚动 | 300 上限 vs 无限 | 防 GC 死亡螺旋 |
| StylePool | 避免重复 ANSI 计算 | 转换缓存 |
| CharPool | 减少 GC 压力 | 字符串缓存 |
| 脏标记 | 只渲染变化的子树 | 冒泡标记 |

## 6. 安全架构分层

```
Layer 1: 配置规则（settings.json）
    │ allow/deny/ask 规则
    ▼
Layer 2: 自动模式分类器
    │ 小模型快速分类
    ▼
Layer 3: Hook 系统
    │ 用户/企业自定义检查
    ▼
Layer 4: 交互式对话框
    │ 用户最终决策
    ▼
Layer 5: 拒绝追踪
    │ 记录被拒绝的操作
    ▼
Layer 6: 工具级验证
    │ 23项安全检查（Bash）
    │ 设备路径封锁（FileRead）
    │ 密钥检测（FileWrite）
    ▼
Layer 7: 投机分类器
    │ 后台自动批准安全操作
    ▼
最终执行或拒绝
```

每一层都是独立的安全边界。即使某一层被绕过，后续层仍然提供保护。这是典型的**纵深防御（Defense in Depth）**策略。

## 7. 架构演进方向

从源码结构可以推断出几个正在进行的演进方向：

### 7.1 多 Agent 扩展

Coordinator 模式、TeamCreate、SendMessage 工具的存在表明多 Agent 协作是核心发展方向。当前的消息路由（本地→UDS→Bridge→文件邮箱）设计为支持从单机到分布式的扩展。

### 7.2 远程执行

DirectConnect、SSH、Bridge 三种远程模式的共存表明远程执行场景在不断扩展。Worktree 隔离和会话持久化为远程长时间运行的 Agent 提供了基础。

### 7.3 企业集成

MDM 支持、远程管理设置、GrowthBook feature flags 和策略限制表明企业部署是重要的目标市场。

### 7.4 开发者体验

Buddy 彩蛋、Vim 模式、自定义 Skill、插件系统和丰富的命令集表明开发者体验是产品差异化的关键。

## 8. 总结

Claude Code 是一个精心设计的工业级软件系统。它的架构核心可以归结为：

1. **一个查询引擎**：AsyncGenerator 状态机，驱动所有 AI 交互
2. **一个工具系统**：30+ 方法的 Tool 接口，统一管理 39+ 工具
3. **一个权限管道**：三路处理器，7 种模式，20+ 事件类型
4. **一个渲染引擎**：自定义 React reconciler，双缓冲 + 虚拟滚动
5. **一个状态树**：150+ 字段的 DeepImmutable AppState，35 行 Store
6. **一套编译系统**：Feature Flag DCE，单代码库四种部署变体

这些组件通过明确的接口边界和横切模式（消息规范化、Context Modifier、AsyncGenerator）组合在一起，形成了一个模块化、安全、高性能的系统。
