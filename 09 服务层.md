## 1. API 客户端（services/api/）

### 1.1 多 Provider 工厂

Claude Code 支持四种 API Provider，通过工厂模式创建统一的客户端接口：

```typescript
function createApiClient(config: ApiConfig): ApiClient {
  switch (config.provider) {
    case 'direct':
      return new DirectApiClient(config);    // 直连 Anthropic API
    case 'bedrock':
      return new BedrockApiClient(config);   // AWS Bedrock
    case 'vertex':
      return new VertexApiClient(config);    // Google Vertex AI
    case 'foundry':
      return new FoundryApiClient(config);   // Anthropic Foundry
  }
}
```

#### Direct（直连）

直接连接 `api.anthropic.com`，使用 Anthropic SDK，支持所有功能：
- 流式输出
- 扩展思考
- 工具使用
- Web 搜索
- 文件上传

#### Bedrock

通过 AWS 的 Bedrock 服务访问 Claude：
- 需要 AWS 凭证（IAM/SSO）
- API 格式差异需要适配层
- 某些 beta 功能不可用
- 区域路由（us-east-1, us-west-2 等）

#### Vertex

通过 Google Cloud 的 Vertex AI 访问 Claude：
- 需要 GCP 凭证（service account/ADC）
- 端点格式不同
- 流式格式差异

#### Foundry

Anthropic 自有的企业部署平台：
- 专用端点
- 高级功能优先可用
- SLA 保证

```typescript
async function autoCompact(
  messages: Message[],
  tracking: AutoCompactTracking

  const tokenCount = countTokens(messages);
  const threshold = getMaxTokens() * 0.9;

  // 未达阈值
  if (tokenCount < threshold) return messages;
```

```typescript
async function sessionMemoryCompact(
  messages: Message[],
  claudeMdContent: string
): Promise<Message[]> {
  // 识别消息中与CLAUDE.md重复的内容
  const duplicates = findDuplicateContent(messages, claudeMdContent);

  // 替换重复内容为引用
  return messages.map(msg => {
    for (const dup of duplicates) {
      if (msg.content.includes(dup.text)) {
        msg = {

      }
    }
    return msg;
  });
```

### 3.2 熔断器机制

```typescript
class CompactCircuitBreaker {
  private failures = 0;
  private readonly maxFailures = 3;

  async execute<T>(fn: () => Promise<T>): Promise<T | null> {
    if (this.failures >= this.maxFailures) {
      return null; // 熔断：不再尝试
    }

    try {
      const result = await fn();
      this.failures = 0; // 成功重置
      return result;
    } catch {
      this.failures++;
      return null;
    }
  }
}
```

### 为什么需要熔断器

压缩操作本身消耗 API 调用（使用 LLM 进行摘要）。如果压缩反复失败（如 API 不可用），持续重试会浪费 token 预算。熔断器在 3 次连续失败后停止尝试，避免无效的 API 消耗。

## 4. OAuth 服务

```typescript
class OAuthService {
  // 构建授权URL
  buildAuthorizationUrl(options: AuthOptions): string {
    const params = new URLSearchParams({
      client_id: options.clientId,
      redirect_uri: options.redirectUri,
      response_type: 'code',
      scope: options.scope,
      code_challenge: options.codeChallenge,
      code_challenge_method: 'S256',
      state: generateRandomState(),
    });
    return `${AUTH_ENDPOINT}?${params}`;
  }

  // Token交换
  async exchangeToken(code: string, codeVerifier: string): Promise<TokenResponse> {
    const response = await fetch(TOKEN_ENDPOINT, {
      method: 'POST',
      headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
      body: new URLSearchParams({
        grant_type: 'authorization_code',
        code,
        redirect_uri: REDIRECT_URI,
        code_verifier: codeVerifier,
      }),
    });
    return response.json();
  }

  // PKCE验证
  generatePKCE(): { verifier: string; challenge: string } {
    const verifier = randomBytes(32).toString('base64url');
    const challenge = createHash('sha256').update(verifier).digest('base64url');
    return { verifier, challenge };
  }

  // Profile提取
  async getProfile(accessToken: string): Promise<UserProfile> {
    const response = await fetch(PROFILE_ENDPOINT, {
      headers: { Authorization: `Bearer ${accessToken}` },
    });
    return response.json();
  }

  // Scope协商
  negotiateScope(requested: string[], available: string[]): string[] {
    return requested.filter(s => available.includes(s));
  }
}
```

### 5.4 预附着缓冲

```typescript
// 在logging sinks初始化之前，事件被缓冲
const preAttachBuffer: AnalyticsEvent[] = [];

function logEvent(event: AnalyticsEvent): void {
  if (!sinksInitialized) {
    preAttachBuffer.push(event);
    return;
  }

  // 正常发送
  for (const sink of sinks) {
    sink.log(event);
  }
}

// sinks初始化后，排空缓冲
function initSinks(): void {
  sinksInitialized = true;
  for (const event of preAttachBuffer) {
    logEvent(event);
  }
  preAttachBuffer.length = 0;
}
```

#### 为什么需要预附着缓冲

启动阶段存在时序问题：
- 某些分析事件在 sinks 初始化之前就产生了（如模块加载时间、Keychain 读取时间）
- 不能丢弃这些早期事件（它们是启动性能分析的关键数据）
- 缓冲后在 sinks 就绪时统一发送

## 6. 插件加载器

```typescript
class PluginLoader {
  private cache = new Map<string, Plugin>();

  async loadPlugins(): Promise<Plugin[]> {
    const pluginDirs = await scanPluginDirectories();
    const plugins: Plugin[] = [];

    for (const dir of pluginDirs) {
      const cached = this.cache.get(dir);
      if (cached && !this.needsRefresh(dir, cached)) {
        plugins.push(cached);
        continue;
      }

      const plugin = await this.loadPlugin(dir);
      this.cache.set(dir, plugin);
      plugins.push(plugin);
    }

    return plugins;
  }

  private needsRefresh(dir: string, cached: Plugin): boolean {
    const stat = fs.statSync(path.join(dir, 'plugin.json'));
    return stat.mtimeMs > cached.loadedAt;
  }
}
```

## 7. Token 估算

```typescript
function estimateTokens(message: Message): number {
  if (typeof message.content === 'string') {
    // 粗略估算：4字符≈1token
    return Math.ceil(message.content.length / 4);
  }

  // 多模态消息
  let total = 0;
  for (const block of message.content) {
    switch (block.type) {
      case 'text':
        total += Math.ceil(block.text.length / 4);
        break;
      case 'image':
        total += estimateImageTokens(block.width, block.height);
        break;
      case 'tool_use':
        total += Math.ceil(JSON.stringify(block.input).length / 4);
        break;
    }
  }
  return total;
}
```

## 8. 设计决策总结

| 决策 | 原因 |
|------|------|
| 多Provider工厂 | 支持企业部署（Bedrock/Vertex） |
| MCP 4种传输 | 覆盖本地/远程/浏览器场景 |
| 5种压缩策略 | 不同场景用不同策略 |
| 熔断器 | 防止压缩失败无限重试浪费API |
| 预附着缓冲 | 不丢失启动阶段的分析事件 |
| 插件热加载 | 开发中修改插件不需要重启 |
| queueMicrotask排空 | 异步日志不阻塞主线程 |
