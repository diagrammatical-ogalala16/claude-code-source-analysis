## 1. 自定义 React Reconciler

Claude Code 没有使用 Ink 的默认 reconciler，而是构建了一套完整的自定义 React reconciler。

### 1.1 createReconciler

```typescript
const reconciler = createReconciler({
  // 50+ 生命周期钩子
  createInstance(type, props) {
    switch (type) {
      case 'ink-box': return new BoxNode(props);
      case 'ink-text': return new TextNode(props);
      case 'ink-virtual-text': return new VirtualTextNode(props);
      case 'ink-link': return new LinkNode(props);
      case 'ink-raw-ansi': return new RawAnsiNode(props);
      default: return new BoxNode(props);
    }
  },

  appendChild(parent, child) {
    parent.appendChild(child);
    parent.markDirty();
  },

  removeChild(parent, child) {
    parent.removeChild(child);
    parent.markDirty();
  },

  commitUpdate(node, updatePayload, type, oldProps, newProps) {
    node.applyProps(newProps);
    node.markDirty();
  },

  // 自动聚焦
  commitMount(node) {
    if (node.props.autoFocus) {
      node.focus();
    }
  },

  // ...更多钩子
});
```

### 1.4 脏标记追踪

```typescript
class DOMNode {
  private dirty = true;

  markDirty() {
    if (!this.dirty) {
      this.dirty = true;
      this.parent?.markDirty();  // 冒泡到父节点
    }
  }

  isDirty(): boolean {
    return this.dirty;
  }

  clearDirty() {
    this.dirty = false;
  }
}
```

只有标记为脏的子树才会重新布局和渲染，避免全量重算。

## 3. 双缓冲系统

### 3.1 Screen 实现

```typescript
class Screen {
  private frontBuffer: ScreenBuffer;
  private backBuffer: ScreenBuffer;

  beginFrame(): void {
    // 交换缓冲区
    [this.frontBuffer, this.backBuffer] = [this.backBuffer, this.frontBuffer];
    this.backBuffer.clear();
  }

  draw(operations: OutputOperation[]): void {
    for (const op of operations) {
      this.backBuffer.apply(op);
    }
  }

  endFrame(): string {
    // 计算差异
    const diff = this.computeDiff(this.frontBuffer, this.backBuffer);
    return diff;
  }
}
```

### 3.2 为什么需要双缓冲

- **防止闪烁**：复杂的重排（消息流 + 工具结果同时更新）如果直接写终端会闪烁
- **最小化输出**：只输出两帧之间的差异，减少终端 I/O
- **原子更新**：一帧要么完全显示要么完全不显示，不会出现半渲染状态

### 3.3 StylePool（ANSI 代码缓存）

```typescript
class StylePool {
  private styles = new Map<number, string>();      // styleId → ANSI码
  private transitionCache = new Map<number, string>(); // 转换缓存

  // 分配/查找样式ID
  internStyle(ansiCode: string): number {
    const id = hash(ansiCode);
    if (!this.styles.has(id)) {
      this.styles.set(id, ansiCode);
    }
    return id;
  }

  // 获取从样式A到样式B的ANSI转换序列
  getTransition(fromId: number, toId: number): string {
    const cacheKey = fromId * 0x100000 + toId;

    if (!this.transitionCache.has(cacheKey)) {
      const transition = computeTransition(
        this.styles.get(fromId)!,
        this.styles.get(toId)!
      );
      this.transitionCache.set(cacheKey, transition);
    }

    return this.transitionCache.get(cacheKey)!;
  }
}
```

#### 为什么需要转换缓存

ANSI 样式切换需要计算两种样式之间的差异（哪些属性需要重置，哪些需要设置）。这个计算涉及字符串解析和比较，在高频渲染中（每帧可能有数百次样式切换）成本显著。

缓存键 `fromId * 0x100000 + toId` 是一个简单的 Cantor 配对变体，用乘法+加法避免碰撞。

### 3.4 CharPool（字符串缓存）

```typescript
class CharPool {
  private pool = new Map<string, string>();

  intern(str: string): string {
    const existing = this.pool.get(str);
    if (existing) return existing;

    this.pool.set(str, str);
    return str;
  }
}
```

终端输出生成大量相同的短字符串（空格、制表符、常见标点）。缓存这些字符串可以显著减少 GC 压力——V8 的 GC 对大量短生命周期字符串尤其敏感。

class HyperlinkPool {

  getOSC8(url: string, text: string): string {
    const key = `${url}:${text}`;
    if (!this.links.has(key)) {
      this.links.set(key, `\x1b]8;;${url}\x1b\\${text}\x1b]8;;\x1b\\`);

}

### 4.2 DECSTBM 硬件滚动优化

```typescript
// 使用终端的滚动区域功能
function hardwareScroll(topLine: number, bottomLine: number, scrollAmount: number): string {
  // 设置滚动区域
  let output = `\x1b[${topLine};${bottomLine}r`;

  // 执行滚动
  if (scrollAmount > 0) {
    // 向下滚动（内容上移）
    output += `\x1b[${bottomLine}H`;
    output += '\n'.repeat(scrollAmount);
  } else {
    // 向上滚动（内容下移）
    output += `\x1b[${topLine}H`;
    output += `\x1b[${-scrollAmount}L`;
  }

  // 恢复全屏滚动区域
  output += `\x1b[r`;
  return output;
}
```

#### 为什么硬件滚动比重渲染快

- **重渲染**：需要重新计算和输出所有可见行（可能数百行）
- **硬件滚动**：终端硬件/仿真器在内部移动行缓冲区，只需几个控制序列

对于滚动大量对话历史的场景，硬件滚动速度快几个数量级。

### 4.3 相对光标移动

```typescript
function moveCursor(fromX: number, fromY: number, toX: number, toY: number): string {
  let output = '';

  // 垂直移动
  const dy = toY - fromY;
  if (dy > 0) output += `\x1b[${dy}B`;      // 下移
  else if (dy < 0) output += `\x1b[${-dy}A`; // 上移

  // 水平移动
  const dx = toX - fromX;
  if (dx > 0) output += `\x1b[${dx}C`;       // 右移
  else if (dx < 0) output += `\x1b[${-dx}D`; // 左移

  return output;
}
```

相对移动比绝对定位（`\x1b[y;xH`）在短距离移动时更高效（更少字符）。

class VirtualScreen {

## 5. 宽字符处理

### 5.1 Emoji 宽度补偿

```typescript
function getCharWidth(char: string): number {
  const codePoint = char.codePointAt(0)!;

  // VS16 (U+FE0F) 将前一个字符变为宽字符
  if (codePoint === 0xFE0F) return 0; // 本身不占宽度

  // Emoji 宽字符
  if (isEmojiPresentation(codePoint)) return 2;

  // CJK 字符
  if (isCJK(codePoint)) return 2;

  // 组合字符
  if (isCombining(codePoint)) return 0;

  return 1;
}
```

### 5.2 Unicode 12-15 符号检测

```typescript
function isEmojiPresentation(codePoint: number): boolean {
  // Unicode 12.0+ 新增的Emoji范围
  if (codePoint >= 0x1FA70 && codePoint <= 0x1FAFF) return true;  // Symbols Extended-A
  if (codePoint >= 0x1F900 && codePoint <= 0x1F9FF) return true;  // Supplemental Symbols

  // Unicode 15.0 新增
  if (codePoint >= 0x1FAE0 && codePoint <= 0x1FAEF) return true;

  // 查表
  return EMOJI_PRESENTATION_TABLE.has(codePoint);
}
```

### 5.3 VS16 处理

```
U+2764 (❤) → 默认文本表示 → 宽度1
U+2764 U+FE0F (❤️) → Emoji表示 → 宽度2
```

一些终端的宽度表是旧版本的 Unicode，会在新 Emoji 上计算错误。Claude Code 内置了最新的宽度表来补偿。

### 6.1 捕获/冒泡阶段

```typescript
function dispatchEvent(event: TerminalEvent, target: DOMNode): void {
  const path = getEventPath(target); // root → ... → target

  // 捕获阶段（从根到目标）
  for (const node of path) {
    node.handleCapture(event);
    if (event.stopped) return;
  }

  // 冒泡阶段（从目标到根）
  for (const node of path.reverse()) {
    node.handleBubble(event);
    if (event.stopped) return;
  }
}
```

## 7. 虚拟滚动

### 7.1 useVirtualScroll Hook

```typescript
function useVirtualScroll(options: VirtualScrollOptions) {
  // Float64Array存储每个item的Y偏移量
  const offsets = useRef(new Float64Array(options.itemCount));

  const OVERSCAN_ROWS = 80;      // 预渲染行数
  const MAX_MOUNTED_ITEMS = 300;  // 最大挂载组件数
  const SLIDE_STEP = 25;          // 每次滑动步长

  // 计算可见范围
  function getVisibleRange(scrollTop: number, viewportHeight: number) {
    // 二分搜索找到第一个可见item
    const startIndex = binarySearch(offsets.current, scrollTop - OVERSCAN_ROWS);
    const endIndex = binarySearch(offsets.current, scrollTop + viewportHeight + OVERSCAN_ROWS);

    // 限制挂载数量
    const mountedCount = endIndex - startIndex;
    if (mountedCount > MAX_MOUNTED_ITEMS) {
      // 滑动窗口
      return slideWindow(startIndex, endIndex, MAX_MOUNTED_ITEMS, SLIDE_STEP);
    }

    return { start: startIndex, end: endIndex };
  }
}
```

### 7.2 为什么用 Float64Array

- **O(1) 二分搜索**：typed array 内存连续，二分搜索利用 CPU cache 局部性
- **缓存友好**：Float64Array 是连续内存，比 JavaScript 数组（对象数组）的缓存命中率高
- **精度**：双精度浮点数避免了大偏移量的精度损失

### 7.3 MAX_MOUNTED_ITEMS = 300 的原因

在测试中发现：
- ~2000 个消息组件同时挂载时，RSS 达到 59GB
- GC 进入"死亡螺旋"——每次 GC 暂停导致更多对象被提升到老年代
- 300 是一个安全上限，在 1GB RSS 以内

### 7.4 SLIDE_STEP = 25

滑动步长控制每帧新挂载的 React fiber 数量：
- 步长太大：一次挂载太多组件导致帧卡顿
- 步长太小：追赶滚动位置太慢，出现空白
- 25 是经验值：在 60fps 目标下，每帧挂载 25 个组件不会导致明显掉帧

### 7.5 高度缓存与缩放

```typescript
// 终端大小变化时，按比例调整缓存的高度
function scaleHeightCache(
  cache: Float64Array,
  oldColumns: number,
  newColumns: number
): void {
  const ratio = newColumns / oldColumns;
  for (let i = 0; i < cache.length; i++) {
    // 文本重排可能改变高度，按列比例估算
    cache[i] = Math.ceil(cache[i] / ratio);
  }
  // 重新计算偏移量
  recalculateOffsets(cache);
}
```
